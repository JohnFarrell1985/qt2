# 用户手册

## 1. 环境准备

### 1.1 系统要求
- Python 3.10+
- PostgreSQL 14+
- 迅投 MiniQMT 客户端 (数据/交易功能)
- [uv](https://docs.astral.sh/uv/) 包管理器

### 1.2 安装

```bash
# 克隆项目
git clone <repo_url> qt
cd qt

# 使用 uv 安装依赖
uv sync

# 开发模式 (含测试依赖)
uv sync --extra dev
```

### 1.3 配置

```bash
# 编辑项目根目录 .env 填写:
#   DATABASE_URL=postgresql://user:pass@host:5432/qt_quant
#   QMT_MINI_PATH=C:/MiniQMT/userdata_mini
#   QMT_ACCOUNT_ID=your_account
```

详细参数说明见 `.env` 文件内注释和本手册第 6 章。

## 2. 快速上手

### 2.1 初始化数据库

```bash
uv run python scripts/init_db.py
```

### 2.2 同步数据

```bash
# 需先启动 MiniQMT 客户端
uv run python scripts/sync_factors.py
```

### 2.3 训练模型

```bash
uv run python scripts/train_model.py
```

### 2.4 启动 API 服务

```bash
uv run uvicorn src.api.main:app --host 0.0.0.0 --port 8012
```

访问 http://localhost:8012/docs 查看 Swagger 文档。

## 3. 核心操作流程

## 5. 常见问题

### Q: QMT 连接失败
确认 MiniQMT 客户端已启动，`QMT_MINI_PATH` 指向正确的 `userdata_mini` 目录。

### Q: 数据库连接失败
检查 `.env` 中的 `DATABASE_URL` 环境变量。

### Q: 因子数据为空
先执行数据同步: `uv run python scripts/sync_factors.py`

### Q: 模型训练数据不足
确保数据库中有足够的历史行情和因子数据 (建议 >= 2年日线)。

### Q: 下载卡住或超时
见第 6 章排障指南。

## 6. 数据下载攻略

> 完整的从零开始到数据入库的操作指南, 涵盖原理、配置、操作步骤和排障。

### 6.1 前置条件

| 项目 | 要求 | 说明 |
|------|------|------|
| MiniQMT 客户端 | 已安装并**保持运行** | xtquant SDK 通过本地进程通信读写数据 |
| 账户类型 | 券商实盘 或 迅投模拟盘 | 免费版即可获取行情/财务数据 |
| 磁盘空间 | ≥ 50 GB (仅日线) / ≥ 500 GB (含1分钟线) | 数据缓存在 MiniQMT 的 `userdata_mini` 目录 |
| 网络 | 稳定连接 | 全量日线约需 30~60 分钟, 分钟线按量级可能需要数小时 |

### 6.2 下载架构概览

```
                        ┌─────────────┐
                        │  用户/API    │
                        └──────┬──────┘
                               ▼
                     ┌──────────────────┐
                     │  DataSyncManager │ ← sync.py 统一调度
                     │  full_sync()     │
                     │  incremental()   │
                     └────────┬─────────┘
                              │
            ┌─────────────────┼─────────────────┐
            ▼                 ▼                 ▼
      MarketDataSync   FinancialDataSync   FactorDataManager
            │                 │                 │
            └────────┬────────┴────────┬────────┘
                     ▼                 ▼
              DownloadEngine     DownloadEngine
              (download_kline)   (download_financial)
                     │                 │
              ┌──────┴──────┐   ┌──────┴──────┐
              │ 分批500只    │   │ 分批500只    │
              │ 暂停2s/批   │   │ 暂停2s/批   │
              │ 异步等待     │   │ 异步等待     │
              │ 失败重试x3  │   │ 失败重试x3  │
              └──────┬──────┘   └──────┬──────┘
                     ▼                 ▼
              xtdata SDK (MiniQMT 本地进程)
                     │
                     ▼
              迅投服务器 (行情中心)
```

### 6.3 关键原理 (务必理解)

#### 6.3.1 先下载, 再获取

xtquant 的 `get_*` 系列接口**只读取本地缓存**, 不会主动去服务器拉取数据。必须先调用对应的 `download_*` 将数据下载到 MiniQMT 本地目录, 然后才能用 `get_*` 读取。

```
错误流程: get_market_data_ex() → 空数据!
正确流程: download_history_data2() → 等待完成 → get_local_data() → 有数据
```

#### 6.3.2 download_history_data2 是异步的

虽然官方文档标注为"同步", 但社区大量实测表明该接口实际是**异步执行**的:
- 调用后立即返回, 后台下载
- 如果立即调用 `get_local_data`, 大概率拿到不完整的数据
- 本平台通过 `threading.Event` + `callback` 机制等待下载真正完成

#### 6.3.3 增量续传

```python
# incrementally=None (默认): xtdata 自动判断
#   - 本地已有数据 → 从最后一条之后继续下载 (断点续传)
#   - 本地无数据   → 从 start_time 开始全量下载
# incrementally=False: 强制删除本地缓存, 全部重新下载
```

#### 6.3.4 批量限制

MiniQMT 单次请求处理能力有限:
- 一次下载 5000 只股票 → 高概率卡死或超时
- 社区最佳实践: **每批 ≤ 500 只, 批间暂停 2~3 秒**
- 本平台已内置此策略, 无需手动分批

### 6.4 .env 下载参数配置

```bash
# ── 核心参数 ──
DL_BATCH_SIZE=500      # 每批股票数量 (推荐 300~500)
DL_BATCH_PAUSE=2.0     # 批间暂停秒数 (推荐 2.0~3.0)
DL_RETRY_COUNT=3       # 失败重试次数
DL_RETRY_DELAY=5.0     # 重试间隔基数 (指数退避: 5s, 10s, 15s)
DL_TIMEOUT=600         # 单批超时 (日线60s够, 1m线建议600s)

# ── 各周期默认起始日期 ──
DL_START_1D=20160101   # 日线: 10年
DL_START_1W=20160101   # 周线: 10年
DL_START_5M=20230101   # 5分钟: 3年
DL_START_15M=20230101  # 15分钟: 3年
DL_START_30M=20230101  # 30分钟: 3年
DL_START_1H=20230101   # 1小时: 3年
DL_START_1M=20250101   # 1分钟: 1年 (数据量巨大!)
DL_START_TICK=20260301 # tick: 仅最近1个月
```

#### 数据量估算

| 周期 | 每天每只 | 5000只 × 1年 | 5000只 × 3年 | 5000只 × 10年 |
|------|----------|-------------|-------------|--------------|
| 日线 | 1条 | 125万 | 375万 | 1250万 |
| 5分钟 | 48条 | 6000万 | 1.8亿 | 6亿 |
| 1分钟 | 240条 | 3亿 | 9亿 | **30亿** |
| tick | ~4800条 | **60亿** | — | — |

> **存储提示**: 日线数据 PostgreSQL 约占 2~5 GB; 1分钟线1年约 50~100 GB; tick 量级不建议入库, 仅按需本地缓存。

### 6.5 操作步骤

#### 步骤一: 首次全量同步 (推荐)

**通过 API:**

```bash
# 仅日线 + 财务 + 因子 (最快, 约 30~60 分钟)
curl -X POST "http://localhost:8012/api/data/sync/full?incremental=true"

# 日线 + 5分钟线 (约 2~4 小时)
curl -X POST "http://localhost:8012/api/data/sync/full?sync_minute=true&minute_periods=5m&incremental=true"

# 日线 + 多周期分钟线 (约 4~8 小时)
curl -X POST "http://localhost:8012/api/data/sync/full?sync_minute=true&minute_periods=5m,15m,1m&incremental=true"
```

**通过 Python:**

```python
from src.data.sync import DataSyncManager

mgr = DataSyncManager()

# 仅日线
results = mgr.full_sync(incremental=True)

# 日线 + 5分钟线
results = mgr.full_sync(sync_minute=True, minute_periods=["5m"], incremental=True)

# 指定起始日期 (覆盖 .env 默认值)
results = mgr.full_sync(start_date="20240101", sync_minute=True, incremental=True)
```

全量同步执行顺序:
1. 基础数据 (板块、节假日、指数权重) — 约 1 分钟
2. 股票列表 + 合约详情 — 约 2 分钟
3. 日线行情 (10批 × 500只) — 约 15~30 分钟
4. 分钟线 (按周期, 数据量大) — 视配置而定
5. 指数日线 (8支) — 约 1 分钟
6. 财务报表 + 指标 — 约 10~20 分钟
7. 因子数据 — 约 5~10 分钟

#### 步骤二: 每日增量同步

```bash
# API (定时任务自动执行, 也可手动触发)
curl -X POST "http://localhost:8012/api/data/sync/incremental?days_back=5"
```

增量模式仅下载最近 N 天数据, 执行时间约 3~5 分钟。

平台内置定时调度: 每日 17:00 自动执行增量同步。

#### 步骤三: 中断后恢复 (断点续传)

如果下载过程中断 (网络断开、MiniQMT 重启等), 只需**重新执行同样的命令**:

```bash
# 再次执行, 引擎会自动跳过已下载的数据
curl -X POST "http://localhost:8012/api/data/sync/full?incremental=true"
```

原理: `incremental=true` 时, xtdata 检测本地已有数据, 自动从断点处继续。

#### 步骤四: 强制重新下载

如果怀疑本地数据损坏, 可强制全量重下:

```bash
curl -X POST "http://localhost:8012/api/data/sync/full?incremental=false"
```

> **注意**: `incremental=false` 会删除本地缓存后重新下载, 耗时较长。

| 场景 | 日线 | 分钟线 | 财务 | 说明 |

### 6.9 排障指南

#### 问题: 下载超时

**现象**: 日志出现 `批次 X 下载超时 (600s)`

**排查**:
1. 检查 MiniQMT 客户端是否仍在运行
2. 检查网络连接 (迅投服务器 → 本机)
3. 尝试增大超时: `.env` 中设置 `DL_TIMEOUT=900`
4. 尝试减小批量: `DL_BATCH_SIZE=200`

#### 问题: 下载卡死 / MiniQMT 无响应

**现象**: 下载进度不再推进, MiniQMT 界面无响应

**排查**:
1. 重启 MiniQMT 客户端
2. 减小批量: `DL_BATCH_SIZE=300`
3. 增大暂停: `DL_BATCH_PAUSE=3.0`
4. 重新执行同步 (自动断点续传)

#### 问题: 限流 / 连接被拒

**现象**: 日志出现连接错误或频繁重试

**排查**:
1. 增大暂停: `DL_BATCH_PAUSE=5.0`
2. 减小批量: `DL_BATCH_SIZE=200`
3. 等待 10~30 分钟后重试
4. 检查是否有其他程序同时访问 MiniQMT

#### 问题: 数据缺失

**现象**: 数据库中某些股票数据为空

**排查**:
1. 检查日志中该股票是否在失败批次中
2. 可能是停牌/退市股票 (本身无数据)
3. 尝试单独下载: `QMTClient().download_history_data("000001.SZ", "1d", "20240101")`
4. 重新全量同步: `incremental=false`

#### 问题: 磁盘空间不足

**现象**: 下载过程报 IO 错误

**排查**:
1. 检查 MiniQMT `userdata_mini` 目录所在磁盘剩余空间
2. 缩短数据范围: 调整 `.env` 中 `DL_START_*`
3. 清理不需要的周期数据 (如不需要 tick)

### 6.10 性能调优建议

| 参数 | 保守配置 | 默认配置 | 激进配置 | 说明 |
|------|----------|----------|----------|------|
| `DL_BATCH_SIZE` | 200 | 500 | 800 | 越大越快但越不稳定 |
| `DL_BATCH_PAUSE` | 5.0 | 2.0 | 1.0 | 越小越快但越容易限流 |
| `DL_RETRY_COUNT` | 5 | 3 | 1 | 网络不稳建议调高 |
| `DL_TIMEOUT` | 900 | 600 | 300 | 1分钟线建议 ≥ 600 |

**首次下载推荐保守配置**, 确认稳定后再逐步放宽。

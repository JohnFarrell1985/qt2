# 用户手册

## 1. 环境准备

### 1.1 系统要求

- Python 3.10+
- PostgreSQL 14+
- 迅投 QMT 终端 (数据/交易功能)
- [uv](https://docs.astral.sh/uv/) 包管理器

### 1.2 安装

```bash
git clone <repo_url> qt
cd qt
uv sync
uv sync --extra dev  # 含测试依赖
```

### 1.3 配置

环境变量按模块拆分到 `env/` 目录下的多个文件，系统启动时自动合并加载。

```bash
# 复制模板后按需修改
cp env/.env.example env/.env

# 或逐个模块配置:
env/
├── .env.example       # 全量模板 (400+ 行, 含所有参数说明)
├── .env.db            # 数据库连接
├── .env.qmt           # QMT 终端路径与账户
├── .env.datacollect   # 数据采集 + LLM 清洗
├── .env.trading       # 交易手续费
├── .env.backtest      # 回测参数
└── .env.sentiment     # 情绪引擎
```

**必填项:**

```bash
# env/.env.db
DATABASE_URL=postgresql://localhost:5432/qt_quant

# env/.env.qmt
QMT_PATH=C:/QMT/userdata_mini
QMT_ACCOUNT_ID=your_account  # 资金账号, 各券商位数不同
```

**优先级:** 系统环境变量 > `env/.env.*` 文件 > 代码默认值

## 2. 快速上手 (10 分钟)

### Step 1: 初始化数据库

```bash
uv run python scripts/init_db.py
```

### Step 2: 启动 API 服务

```bash
uv run uvicorn src.api.main:app --host 0.0.0.0 --port 8012
```

### Step 3: 初始化标的池

```bash
curl -X POST http://localhost:8012/strategy/pools/init-builtin
```

### Step 4: 同步数据 (首次约 30~60 分钟)

```bash
curl -X POST "http://localhost:8012/api/data/sync/full?incremental=true"
```

### Step 5: 获取操作清单

```bash
curl -X POST http://localhost:8012/strategy/execute \
  -H "Content-Type: application/json" \
  -d '{"total_capital": 500000, "available_cash": 500000, "holdings": []}'
```

## 3. 策略使用指南

### 3.1 我该用哪个策略?

根据你的投资风格和市场判断选择：

| 你的情况 | 推荐策略 | 原因 |
|---------|---------|------|
| 新手, 想简单开始 | 双低可转债 | 有债底保护, 亏损有限 |
| 稳健型, 追求分红 | 低波红利 | 低风险, 适合长期持有 |
| 看好大盘趋势 | 动量 + 均线突破 | 趋势追随, 赚趋势钱 |
| 大盘横盘震荡 | 网格交易 + 双低转债 | 区间操作, 赚波动钱 |
| 想系统化投资 | 多因子打分 | 多维度评估, 系统性强 |
| 对技术有信心 | LightGBM ML | 非线性因子组合, 收益潜力最高 |
| 不知道选什么 | 全部开启, 让仲裁器决定 | 投票机制自动选最优 |

### 3.2 策略组合推荐

#### 保守型组合 (月预期收益 0.5%~1.5%)

```
策略: 双低可转债 (50%) + 低波红利 (50%)
标的: 沪深转债 + 沪深300
适合: 熊市/不确定市场
```

#### 均衡型组合 (月预期收益 1%~3%)

```
策略: 多因子打分 (40%) + 动量 (30%) + 低波红利 (30%)
标的: 中证500 + 沪深300
适合: 温和牛市
```

#### 进攻型组合 (月预期收益 2%~5%, 回撤也大)

```
策略: LGB ML (40%) + 动量 (30%) + 行业轮动 (30%)
标的: 沪深A股 / 中证1000
适合: 强势牛市
```

## 4. 宏观环境管理

### 4.1 什么时候切换宏观状态

系统不会自动判断宏观环境 (这需要人的判断)。建议每周末花 5 分钟评估：

| 观察指标 | 牛市信号 | 熊市信号 |
|---------|---------|---------|
| 上证指数 | 均线多头排列 | 均线空头排列 |
| 成交量 | 万亿以上 | 5000 亿以下 |
| 北向资金 | 持续净流入 | 持续净流出 |
| 新高/新低比 | 新高多 | 新低多 |
| 个人感觉 | "到处是机会" | "怎么选都亏" |

### 4.2 切换操作

```bash
# 判断进入牛市
PUT /strategy/macro/state?state_key=bull_moderate&determined_by=weekly_review

# 判断进入熊市
PUT /strategy/macro/state?state_key=bear_mild&determined_by=weekly_review
```

切换后系统自动:
- 调整仓位倍数 (牛市 100%~120%, 熊市 30%~50%)
- 选择匹配的策略
- 回避不适合的策略

### 4.3 六种状态快速参考

| 状态 | state_key | 仓位 | 核心策略 |
|------|-----------|------|---------|
| 强势牛市 | `bull_strong` | 120% | 动量 + 行业轮动 |
| 温和牛市 | `bull_moderate` | 100% | 动量 + 多因子 + ML |
| 震荡 | `range_bound` | 80% | 网格 + 双低 + 低波 |
| 温和熊市 | `bear_mild` | 50% | 低波 + 双低 |
| 深度熊市 | `bear_deep` | 30% | 双低 (极轻仓) |
| 底部复苏 | `recovery` | 90% | 反转 + 双低 + ML |

## 6. 数据下载攻略

> 完整的从零开始到数据入库的操作指南, 涵盖原理、配置、操作步骤和排障。

### 6.1 前置条件

| 项目 | 要求 | 说明 |
|------|------|------|
| QMT 终端 | 已安装并**保持运行** | xtquant SDK 通过本地进程通信读写数据 |
| 账户类型 | 券商实盘 或 迅投模拟盘 | 免费版即可获取行情/财务数据 |
| 磁盘空间 | ≥ 50 GB (仅日线) / ≥ 500 GB (含1分钟线) | 数据缓存在 QMT 安装目录的 `userdata_mini` 目录 |
| 网络 | 稳定连接 | 全量日线约需 30~60 分钟, 分钟线按量级可能需要数小时 |

### 6.2 下载架构概览

```
                        ┌─────────────┐
                        │  用户/API    │
                        └──────┬──────┘
                               ▼
                     ┌──────────────────┐
                     │  DataSyncManager │ ← sync.py 统一调度
                     │  full_sync()     │
                     │  incremental()   │
                     └────────┬─────────┘
                              │
            ┌─────────────────┼─────────────────┐
            ▼                 ▼                 ▼
      MarketDataSync   FinancialDataSync   FactorDataManager
            │                 │                 │
            └────────┬────────┴────────┬────────┘
                     ▼                 ▼
              DownloadEngine     DownloadEngine
              (download_kline)   (download_financial)
                     │                 │
              ┌──────┴──────┐   ┌──────┴──────┐
              │ 分批500只    │   │ 分批500只    │
              │ 暂停2s/批   │   │ 暂停2s/批   │
              │ 异步等待     │   │ 异步等待     │
              │ 失败重试x3  │   │ 失败重试x3  │
              └──────┬──────┘   └──────┬──────┘
                     ▼                 ▼
              xtdata SDK (QMT 本地进程)
                     │
                     ▼
              迅投服务器 (行情中心)
```

### 6.3 关键原理 (务必理解)

#### 先下载, 再获取

xtquant 的 `get_*` 系列接口**只读取本地缓存**, 不会主动去服务器拉取数据。必须先调用 `download_*` 将数据下载到本地, 然后才能用 `get_*` 读取。

```
错误流程: get_market_data_ex() → 空数据!
正确流程: download_history_data2() → 等待完成 → get_local_data() → 有数据
```

#### download_history_data2 是异步的

虽然官方文档标注为"同步", 但社区大量实测表明该接口实际是**异步执行**的。本平台通过 `threading.Event` + `callback` 机制等待下载真正完成。

#### 增量续传

```python
# incrementally=None (默认): xtdata 自动判断
#   - 本地已有数据 → 从最后一条之后继续下载 (断点续传)
#   - 本地无数据   → 从 start_time 开始全量下载
```

### 6.4 操作步骤

#### 首次全量同步

```bash
# 仅日线 + 财务 + 因子 (最快, 约 30~60 分钟)
curl -X POST "http://localhost:8012/api/data/sync/full?incremental=true"

# 日线 + 5分钟线 (约 2~4 小时)
curl -X POST "http://localhost:8012/api/data/sync/full?sync_minute=true&minute_periods=5m&incremental=true"
```

#### 每日增量同步

```bash
curl -X POST "http://localhost:8012/api/data/sync/incremental?days_back=5"
```

平台内置定时调度: 每日 17:00 自动执行增量同步。

#### 中断后恢复

重新执行同样的命令即可 (自动断点续传):

```bash
curl -X POST "http://localhost:8012/api/data/sync/full?incremental=true"
```

### 6.5 数据量估算

| 周期 | 每天每只 | 5000只 × 1年 | 5000只 × 3年 | 5000只 × 10年 |
|------|----------|-------------|-------------|--------------|
| 日线 | 1条 | 125万 | 375万 | 1250万 |
| 5分钟 | 48条 | 6000万 | 1.8亿 | 6亿 |
| 1分钟 | 240条 | 3亿 | 9亿 | **30亿** |

### 6.6 排障指南

| 问题 | 现象 | 解决方案 |
|------|------|---------|
| 下载超时 | `批次 X 下载超时` | 增大 `DL_TIMEOUT`、减小 `DL_BATCH_SIZE` |
| QMT 终端卡死 | 进度不动 | 重启 QMT 终端, 重新同步 (自动续传) |
| 限流 | 频繁重试 | 增大 `DL_BATCH_PAUSE=5.0` |
| 数据缺失 | 某些股票为空 | 可能停牌/退市, 尝试单独下载 |
| 磁盘不足 | IO 错误 | 缩短数据范围 (调整 `DL_START_*`) |

## 7. 数据采集 (datacollect)

> 多源异步采集引擎，支持 9 种数据源自动降级。与第 6 节的 QMT 数据下载不同，datacollect 面向**新闻/舆情/非行情类数据**的采集。

### 7.1 架构概览

```
FallbackDispatcher (按数据类型自动降级)
    ↓
DataSourceRegistry ← data_sources.json (数据源元数据)
    ↓
┌────────────────────────────────────────────────────────┐
│ BaostockCollector    证券宝 (免费, 日线/分钟线)        │
│ AkshareCollector     AKShare (东财链路, 多品类)        │
│ TushareCollector     Tushare Pro (需 token)            │
│ AdataCollector       多源 A 股                         │
│ PytdxCollector       通达信协议                        │
│ EastmoneyCollector   东财 HTTP 直连                    │
│ YfinanceCollector    全球市场 (yfinance)               │
│ SinaGlobalCollector  新浪全球行情                      │
│ NewsRssCollector     财经 RSS (feedparser)             │
└────────────────────────────────────────────────────────┘
    ↓
AsyncCollectEngine (异步并发编排)
    ↓
SmartHttpClient / AsyncSmartHttpClient (六层反爬)
    ├── TLS 指纹伪装 (curl_cffi)
    ├── 令牌桶限流
    ├── 指数退避重试
    ├── 熔断器 (CircuitBreaker)
    ├── 反爬哨兵 (AntiCrawlSentinel)
    └── 死信队列 (DeadLetterDAO)
```

### 7.2 配置

编辑 `env/.env.datacollect`:

```bash
# 限流 (各数据源独立)
DATACOLLECT_AKSHARE_RATE=0.15      # 每秒令牌 (0.15 ≈ 每 7 秒 1 次)
DATACOLLECT_AKSHARE_BURST=3        # 突发上限
DATACOLLECT_BAOSTOCK_RATE=5.0
DATACOLLECT_TUSHARE_RATE=0.8       # 免费用户 50次/分
TUSHARE_TOKEN=                     # 留空则禁用 tushare

# HTTP 策略
DATACOLLECT_MAX_RETRIES=5          # 最大重试
DATACOLLECT_REQUEST_TIMEOUT=30     # 超时秒数
DATACOLLECT_IMPERSONATE=chrome     # 浏览器指纹
DATACOLLECT_PROXY_URL=             # HTTP 代理 (留空不用)

# 熔断器
DATACOLLECT_CB_FAILURE_THRESHOLD=5 # 连续失败 5 次触发熔断
DATACOLLECT_CB_COOLDOWN_SEC=300    # 熔断冷却 5 分钟
```

### 7.3 使用方式

#### 检查数据源可用性

```bash
uv run python -m src.datacollect.dispatcher check
```

输出各数据源的连通状态和响应时间。

#### 在代码中使用

```python
from src.datacollect import FallbackDispatcher, DataSourceRegistry

dispatcher = FallbackDispatcher()
# 按降级链自动选择可用数据源
result = dispatcher.fetch("daily_kline", code="000001.SZ", start="2025-01-01")
```

#### 通过 API 触发

```bash
# QMT 行情数据同步 (走 DownloadEngine, 非 FallbackDispatcher)
POST /api/data/sync/full?incremental=true

# 增量同步 (每日 17:00 自动执行)
POST /api/data/sync/incremental?days_back=5
```

#### AKShare 独立同步脚本

```bash
# 股票列表 + 日线 + 指数 全量同步
uv run python -m src.data.akshare_sync all

# 仅日线增量 (近 N 天)
uv run python -m src.data.akshare_sync daily --days-back 5

# 仅指数数据
uv run python -m src.data.akshare_sync index
```

#### AKShare 财务数据同步

```bash
# ETF 列表 + 日线 + 财报 + 指标
uv run python -m src.data.akshare_financial_sync all

# 仅财务报表
uv run python -m src.data.akshare_financial_sync report

# 仅 ETF 日线
uv run python -m src.data.akshare_financial_sync etf_daily
```

### 7.4 降级链机制

当首选数据源失败时，系统按 `data_sources.json` 中定义的降级链自动切换:

```
daily_kline: akshare → baostock → tushare → adata → pytdx → eastmoney
news:        news_rss → akshare
global:      yfinance → sina_global
```

每级失败后触发: 重试 → 熔断检查 → 降级至下一源 → 全部失败则写入死信队列等待后续重试。

### 7.5 排障

| 问题 | 解决方案 |
|------|---------|
| akshare 频繁 429 | 降低 `DATACOLLECT_AKSHARE_RATE` 至 `0.1` |
| tushare 无数据 | 检查 `TUSHARE_TOKEN` 是否已填且有效 |
| 全部数据源熔断 | 等待 `CB_COOLDOWN_SEC` 秒后自动恢复，或检查网络 |
| 东财反爬封禁 | 配置 `DATACOLLECT_PROXY_URL`，或增大限流间隔 |
| RSS 无新闻 | 检查 `DATACOLLECT_RSSHUB_BASE_URL` 是否可访问 |

## 8. 数据清洗 (dataclean)

> 基于 LLM 的结构化数据抽取引擎，将非结构化文本 (新闻/公告/研报) 转为 Pydantic Schema。

### 8.1 架构概览

```
原始文本 / 新闻数据
    ↓
SentimentCleaner (LLM 模式)
    ├── instructor + OpenAI 兼容 SDK
    ├── Pydantic Schema 强制结构化输出
    └── 自动重试 (tenacity)
    ↓
LLMClient (三级降级)
    ├── Tier 1: DeepSeek Chat      (首选, 性价比最高)
    ├── Tier 2: DeepSeek Reasoner  (复杂推理)
    └── Tier 3: Qwen               (兜底)
    ↓ 全部失败
RuleCleaner (规则兜底)
    ├── 关键词匹配
    └── 正则抽取
    ↓
结构化数据 (SentimentExtraction)
    ├── news_sentiment_score  (-1.0 ~ 1.0)
    ├── key_events[]          (事件列表)
    ├── hot_stocks[]          (涉及个股)
    └── market_impact         (市场影响等级)
```

### 8.2 配置

编辑 `env/.env.datacollect` 的 LLM 部分:

```bash

```

### 8.3 使用方式

#### 在代码中使用

```python
from src.common.config import settings
from src.dataclean.llm_client import LLMClient
from src.dataclean.cleaners.sentiment_cleaner import SentimentCleaner
from src.dataclean.cleaners.rule_cleaner import RuleCleaner

# 初始化
llm_client = LLMClient(settings.dataclean)
cleaner = SentimentCleaner(llm_client=llm_client)

# LLM 清洗
try:
    result = cleaner.clean("央行宣布降准0.5个百分点，释放长期资金约1万亿元")
    print(result.news_sentiment_score)  # 0.7 (利好)
    print(result.key_events)            # [KeyEvent(event="央行降准", ...)]
except Exception:
    # 降级到规则清洗
    rule_cleaner = RuleCleaner()
    result = rule_cleaner.clean("央行宣布降准0.5个百分点")
```

#### 可用的 Cleaner

| Cleaner | 说明 | 需要 LLM |
|---------|------|---------|
| `SentimentCleaner` | 情绪/事件/个股抽取 | 是 |
| `RuleCleaner` | 关键词 + 正则匹配 | 否 |
| `PassthroughCleaner` | 直通 (DataFrame/dict) | 否 |

#### 可用的 Schema

| Schema | 输出字段 | 说明 |
|--------|---------|------|
| `SentimentExtraction` | `news_sentiment_score`, `key_events[]`, `hot_stocks[]`, `market_impact` | 新闻情绪分析 |

> 更多 Schema (个股事件/风险预警) 在 P1-12~15 中规划，尚未实现。

### 8.4 LLM 降级策略

```
                  ┌─── DeepSeek Chat (首选)
LLM_PROVIDER      │     ↓ 失败
  = deepseek ────►├─── Qwen (兜底)
                  │     ↓ 失败
                  └─── AllProvidersFailedError

                  ┌─── Qwen (首选)
LLM_PROVIDER      │     ↓ 失败
  = qwen    ────►├─── DeepSeek Chat (兜底)
                  │     ↓ 失败
                  └─── AllProvidersFailedError

use_reasoner      ┌─── DeepSeek Reasoner
  = True   ────►├─── DeepSeek Chat
                  ├─── Qwen
                  └─── AllProvidersFailedError
```

业务层捕获 `AllProvidersFailedError` 后降级到 `RuleCleaner`。

### 11.2 E2E 端到端测试

E2E 测试按模块分为三个子目录，使用独立的 PostgreSQL schema 隔离，不影响生产数据。

```bash
# ── 按模块运行 ──

uv run pytest tests/e2e/ -v                    # 全部 E2E (含 QMT)
```

**E2E 测试目录结构:**

```
tests/e2e/
├── conftest.py              # 共享 fixtures (DB schema 隔离, TestClient)
├── fixtures/                # 合成数据工厂 (50 stocks × 252 days)
├── api/                     # API E2E — 策略/回测/ML/数据查询/系统健康
├── datacollect/             # 数据采集 E2E — 数据源连通性/采集落盘/降级链
└── qmt/                     # QMT 终端 E2E — 连接/行情/板块/财务/交易
```

**数据采集 E2E 设计原则:**

- **2 分钟硬超时**: 任何数据源 120 秒内无数据返回即判定为不可用，彻底放弃
- **轻量下载**: 每个测试只取 1-2 只股票 / 5 天数据，不触发限流
- **网络容错**: akshare 代理波动时自动 skip，不阻塞测试套件
- **schema 隔离**: 使用 `datacollect_e2e` 独立 schema，测试后自动清理

## 12. 常见问题

### Q: QMT 连接失败
确认 QMT 终端已启动，`QMT_PATH` 指向正确的 `userdata_mini` 目录。

### Q: 数据库连接失败
检查 `.env` 中的 `DATABASE_URL`。

### Q: 因子数据为空
先执行数据同步: `POST /api/data/sync/full?incremental=true`

### Q: 模型训练数据不足
确保数据库中有足够的历史行情和因子数据 (建议 >= 2 年日线)。

### Q: 操作清单为空
可能原因: (1) 宏观状态未设置; (2) 没有激活的策略分配; (3) 数据不足; (4) 所有信号被仲裁器过滤。
检查: `GET /strategy/plan` 查看当前执行计划。

### Q: 止损/止盈没触发
检查 `holdings` 中的 `profit_pct` 和 `hold_days` 是否正确传入。`PositionMonitor` 依赖这些字段判断是否触发风控。

### Q: 实盘交易没执行
确认 `.env` 中 `TRADING_PAPER_MODE=false`，并且代码中调用了 `confirm_live_mode()`。

### Q: LLM 清洗返回空结果
检查: (1) `env/.env.datacollect` 中 `DEEPSEEK_API_KEY` 或 `QWEN_API_KEY` 是否有效;
(2) 运行 `uv run pytest tests/e2e/dataclean/ -v` 验证 API 连通性;
(3) 若两个 API 都失败，业务层需 fallback 到 `RuleCleaner`。

### Q: 数据采集全部熔断
所有数据源连续失败 5 次后触发熔断，冷却 5 分钟后自动恢复。检查网络连接和代理设置。
可通过 `uv run python -m src.datacollect.dispatcher check` 查看各源状态。

### Q: akshare 数据与 QMT 不一致
akshare (东财链路) 和 QMT (迅投服务器) 数据来源不同，日线收盘价可能有微小差异 (复权方式不同)。
以 QMT 为准 (实盘交易用 QMT 下单)，akshare 作为备选数据源。

### Q: 回测 CLI 报 "no data"
确认已同步目标股票的历史数据: `uv run python -m src.backtest.cli info 000001`。
若无数据，先执行: `POST /api/data/sync/full?incremental=true`。

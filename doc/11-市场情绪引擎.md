# 市场情绪引擎 (src/sentiment/)

## 概述

市场情绪引擎是整个量化平台的**大脑中枢**——它负责从多个数据源收集市场情报，计算综合情绪指数，自动判断宏观环境状态，并为每个策略选择对应的参数配置。

**核心目标**: 将当前系统中纯手动的宏观状态判断，升级为"数据驱动建议 + 人工确认"的半自动模式。

```
  多源情报采集 (5 层数据源)
         │
  情绪特征工程 (滚动/Z-score/差分/合成)
         │
  合成情绪指数 (-1 恐慌 ~ +1 贪婪)
         │
  宏观状态分类器 (规则引擎)
         │  建议: "当前应切换到震荡市"
         │
  人工确认 or 全自动
         │
  策略参数 Profile 切换
         │  牛市参数集 / 震荡参数集 / 熊市参数集
         │
  StrategyOrchestrator (动态参数执行)
```

---

## 设计原则

1. **不依赖单一数据源** — 任何一个源挂了都不影响整体判断
2. **零成本优先** — 从已有数据库计算的量价情绪是核心骨架
3. **避免限流** — 每日仅 3-4 次定时采集，总请求 < 25 次/天
4. **可扩展** — 用户可通过数据源注册表自由添加/禁用数据源
5. **半自动** — 系统建议宏观状态，人可确认或覆盖
6. **LLM 清洗优先** — 非结构化数据由大模型清洗为标准 JSON 后再入库

## 技术选型: 为什么不用 Flink / MongoDB

在正式进入架构设计之前，先回答两个关键技术选型问题，避免过度工程化。

### 为什么不需要 Apache Flink

Flink 是分布式流计算引擎，专为高频实时处理设计:

| 维度 | Flink 适用场景 | qt-quant 实际情况 |
|------|---------------|-------------------|
| 数据频率 | 毫秒级 tick，Level-2 逐笔成交 | **日频，每天采集 3-4 次** |
| 吞吐量 | 10,000+ 事件/秒 | **每次 1-5 条数据** |
| 延迟要求 | <100ms 实时响应 | **分钟级甚至小时级都够** |
| 部署要求 | Kafka + Flink + ZooKeeper 集群 | **单机 Docker Compose** |
| 运维成本 | 需要专人运维分布式系统 | **散户一个人维护** |
| 内存消耗 | Flink TaskManager 4-8GB | **服务器资源有限** |
| 适用公司 | 券商自营、百亿私募 | **个人散户量化** |

**结论**: 引入 Flink 是典型的杀鸡用牛刀，增加部署复杂度和运维负担，但收益为零。

**替代方案: APScheduler (Python 定时调度器)**

```
APScheduler (单进程, <10MB 内存)
    │
    ├── 09:00  → run_collect("pre_market")    # 采集+清洗+入库
    ├── 12:00  → run_collect("midday")        # 采集+清洗+入库
    ├── 15:30  → run_collect("post_market")   # 采集+清洗+入库
    └── 20:00  → run_collect("evening")       # 采集+清洗+入库
          │
          ├── 每次执行: 调 akshare 接口 (3-5 秒)
          ├── 接收 OpenClaw 推送 (被动, 0 秒)
          ├── LLM 清洗非结构化数据 (1-3 秒)
          └── 写入 PostgreSQL (< 1 秒)
          
          总耗时: < 10 秒/次, 每日 < 1 分钟
```

APScheduler 内置在 FastAPI 进程中，随主服务启动，无需额外部署任何组件。

> **什么时候才需要 Flink?**
> 当你的策略升级到 **分钟线实时交易**、需要实时处理 **Level-2 逐笔委托/成交** 数据时（日均 30GB+），才需要考虑 Kafka + Flink 流处理架构。目前日频阶段完全不需要。

### 为什么不需要 MongoDB，用 PostgreSQL JSONB

| 维度 | PostgreSQL JSONB | MongoDB |
|------|-----------------|---------|
| **已有基础** | **项目已在用 PostgreSQL** ✅ | 需要新装一个数据库 ❌ |
| 部署复杂度 | 零额外成本 | Docker 多一个容器 + 持久化卷 |
| 运维负担 | 无 (已有) | 多一套备份/监控/升级 |
| SQL 查询 | **标准 SQL + JSONB 操作符混合** | 只能用 MQL (MongoDB 查询语言) |
| 事务支持 | **完整 ACID 事务** | 有限的多文档事务 |
| JOIN 能力 | **可以和 stock_daily 等表 JOIN** ✅ | 跨库 JOIN 极痛苦 |
| 索引 | GIN 索引 (JSONB 内部查询加速) | 原生索引 |
| 数据量 | 日频数据量极小，PostgreSQL 轻松应对 | MongoDB 优势在海量文档 |
| Python 生态 | SQLAlchemy 原生支持 `JSONB` 类型 | 需要额外引入 pymongo/motor |
| 学习成本 | SQL 通用技能 | 需要学 MQL 语法 |

**结论**: PostgreSQL JSONB 完美平替 MongoDB。JSONB 让同一张表中**结构化字段和半结构化字段共存**，兼具关系型数据库和文档数据库的优势。

### JSONB 核心概念 (1 分钟入门)

```
传统关系型:  每列固定类型, 每行固定结构
    trade_date  DATE,  ad_ratio FLOAT,  north_flow FLOAT  ← 全部固定

文档型 (MongoDB):  每行是一个 JSON 文档, 格式自由
    {"trade_date": "2026-04-03", "ad_ratio": 1.72, "events": [...]}

PostgreSQL JSONB:  两者混合 ← 我们用的
    trade_date  DATE,          ← 固定字段 (高频查询)
    ad_ratio    FLOAT,         ← 固定字段
    key_events  JSONB,         ← 半结构化 (格式可变)
    hot_stocks  JSONB,         ← 半结构化
    extra       JSONB          ← 任意扩展
```

**设计原则**:

| 字段类型 | 用 FLOAT/INT/VARCHAR | 用 JSONB |
|---------|---------------------|----------|
| 适用场景 | 格式固定、需要聚合/排序/JOIN | 格式多变、嵌套结构、扩展性 |
| 查询速度 | 最快 (B-tree 索引) | 较快 (GIN 索引) |
| 适合什么 | ad_ratio, north_flow, composite_index | key_events, hot_sectors, hot_stocks |
| 是否用于 LGB | ✅ 直接取值 | ❌ 需要先提取为数值 |

## 数据流架构: 三段式管道 (核心设计)

### 问题: 多源数据格式混乱

情报采集面临的数据格式:

| 数据源 | 原始格式 | 结构化难度 |
|--------|---------|-----------|
| akshare 接口 | pandas DataFrame | 低 (已结构化) |
| 财联社电报 | 短文本+时间 | 低 |
| 新闻聚合 (华尔街见闻/36氪) | 长文本+HTML 片段 | 高 |
| 雪球热帖 | 用户帖子+评论+点赞数 | 中 |
| 同花顺网页 | HTML 表格 | 中 |
| 金投网 | HTML 数据页 | 中 |
| Tavily 搜索结果 | Markdown 摘要+URL | 中 |
| OpenClaw skill 输出 | 自然语言段落 | **高** |

### 方案对比

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| **A: 原始数据全部入 PostgreSQL** | 可追溯、可重放 | 多格式难建表、浪费存储、查询慢 | 高频交易/合规审计 |
| **B: 内存/缓存处理后丢弃** | 快、无存储开销 | 不可追溯、出错无法排查 | 实时流式系统 |
| **C: LLM 清洗 → 结构化 JSON → 入库** | 格式统一、可追溯、成本极低 | 依赖 LLM API | **日频散户量化 (推荐)** |

### 为什么让 LLM 做清洗而不是写代码解析

| 维度 | 传统代码解析 | LLM 清洗 |
|------|------------|---------|
| 格式变化适应性 | 差 (网站改版就挂) | **强** (语义理解) |
| 新数据源接入 | 每个写专用 parser | 改一句 prompt |
| 多语言混合 | 需要分别处理 | 天然支持中英混合 |
| 情感分析 | 需要额外 NLP 模型 | **内置能力** |
| 去噪/去广告 | 复杂规则 | LLM 自然理解 |
| 实体识别 (股票/板块) | 需要 NER 模型 | **内置能力** |
| 成本 | 开发维护高 | API 调用低 |
| 延迟 | <1s | 1-3s |
| 确定性 | 100% 确定 | 95%+ (需 schema 校验) |

**对于日频散户量化 (每日 3-4 次采集)**，LLM 清洗是最优选择。

### LLM 选型与成本估算

| 模型 | 输入价格 | 输出价格 | JSON 模式 | 推荐场景 |
|------|---------|---------|----------|---------|
| **DeepSeek V3** | ¥2/百万 token | ¥8/百万 token | 支持 `json_object` | **首选: 最便宜** |
| Qwen-Turbo | ¥0.3/百万 token | ¥0.6/百万 token | 支持 | 备选: 阿里云生态 |
| Qwen-Max | ¥2/百万 token | ¥6/百万 token | 支持 | 复杂分析 |
| Qwen3.5-Omni | <¥0.8/百万 token | — | 支持 | 最新最便宜 |

**每日成本估算**:

```
每次采集: ~5000 字原始文本 ≈ 3000 tokens
每日 4 次采集 = 12000 input tokens + 3000 output tokens
DeepSeek V3 费用 = 12000/1M × ¥2 + 3000/1M × ¥8 = ¥0.048/天
每月费用 ≈ ¥1.5 (基本免费)
```

### LLM 清洗的两种执行位置

#### 方案 C1: 在 OpenClaw 端清洗 (推荐)

```
OpenClaw (已有 LLM)
    │
    ├── skills 收集原始数据
    ├── OpenClaw 自身的 LLM 清洗为 JSON
    └── POST 结构化 JSON → qt-quant
```

**优点**: OpenClaw 本身就是 AI Agent，自带 LLM 推理能力。它收集完数据后直接理解和结构化，一步到位。不需要 qt-quant 额外调 LLM API。

**实现**: OpenClaw Cron Job + Webhook Delivery (2026.2.17 已支持):

```bash
# OpenClaw 定时任务: 每日 09:00 采集+清洗+推送
openclaw cron add "0 9 * * 1-5" \
  "使用 @news-aggregator @market-news @akshare-a-stock 收集今日A股市场信息,
   汇总为标准JSON格式 POST 到 http://qt-api:8012/api/sentiment/ingest" \
  --name sentiment-premarket \
  --deliver webhook \
  --webhook-url "http://qt-api:8012/api/sentiment/ingest"
```

#### 方案 C2: 在 qt-quant 端清洗

```
OpenClaw
    │
    ├── skills 收集原始数据
    └── POST 原始文本 → qt-quant
                           │
                    qt-quant 调 DeepSeek/Qwen API 清洗
                           │
                    结构化 JSON → 入库
```

**优点**: qt-quant 完全控制清洗逻辑，可以精确定义 JSON Schema。
**缺点**: 需要 qt-quant 额外集成 LLM API (openai SDK)。

#### 推荐: C1 为主 + C2 兜底

- OpenClaw 能搞定的 → C1 (新闻、舆情、综合分析)
- akshare 结构化数据 → 直接入库 (不需要 LLM)
- 量价情绪 → 本地 DB 计算 (不需要 LLM)
- OpenClaw 推送失败时 → C2 兜底 (qt-quant 自己调 DeepSeek 处理)

### LLM 清洗 Prompt 模板

```
System: 你是A股市场数据分析师。将以下市场信息提取为标准JSON。

要求:
1. news_sentiment_score: 整体新闻情绪 -1.0(极度悲观)到+1.0(极度乐观)
2. hot_sectors: 被提及最多的行业板块, 最多5个
3. key_events: 重要事件, 含 event(描述), impact(positive/negative/neutral), magnitude(high/medium/low)
4. hot_stocks: 被频繁讨论的个股, 含 code(代码), reason(原因), sentiment(-1~1)
5. market_mood_text: 一句话市场总结
6. 过滤广告、不相关内容
7. 去重: 相同事件只保留一条

输出严格遵循以下JSON Schema, 不要添加额外字段:
{
  "news_sentiment_score": number,
  "hot_sectors": [string],
  "key_events": [{"event": string, "impact": string, "magnitude": string}],
  "hot_stocks": [{"code": string, "reason": string, "sentiment": number}],
  "gold_price_usd": number | null,
  "crude_oil_usd": number | null,
  "fx_usdcny": number | null,
  "market_mood_text": string
}
```

### 数据存储策略总结

| 数据类型 | 存储位置 | 说明 |
|---------|---------|------|
| 量价情绪 (计算结果) | `sentiment_daily` 表 | 每日一行, 结构化数值 |
| akshare 结构化数据 | `sentiment_daily` 表 | 北向/融资等直接入对应字段 |
| OpenClaw 推送 (已清洗 JSON) | `sentiment_daily` 表 + `ingest_log` | JSON 字段入汇总, 原文入日志 |
| 原始采集文本 | `sentiment_ingest_log.raw_data` (JSONB) | 可追溯, 可重放 |
| 合成指标 / 分类结果 | `sentiment_daily` 表 | 计算后回写 |
| LLM 清洗前的原始文本 | **不入主表** | 仅存日志表, 按需保留 30 天 |

```
结论:
- 结构化数据 (akshare/计算): 直接入 sentiment_daily → 不需要 LLM
- 非结构化数据 (新闻/舆情): LLM 清洗 → JSON → 入 sentiment_daily
- 原始文本: 存 ingest_log (JSONB) → 可追溯但不参与计算
- 不需要 Redis/缓存层 (日频场景, PostgreSQL 足够)
```

## 五层数据采集架构

### Layer 1: 量价情绪 (零成本, 最高优先级)

从已有 PostgreSQL `stock_daily` 表截面计算，**不产生任何网络请求**。

| 指标 | 计算方式 | 含义 |
|------|---------|------|
| 涨跌比 (A/D Ratio) | 上涨家数 / 下跌家数 | >1.5 乐观, <0.5 恐慌 |
| 涨停家数 | 涨幅 >= 9.8% 的股票数 | 市场赚钱效应 |
| 跌停家数 | 跌幅 <= -9.8% 的股票数 | 市场亏钱效应 |
| 炸板率 | 曾涨停后打开的 / 总涨停 | 封板质量 |
| 创 60 日新高家数 | close = max(close, 60d) | 趋势强度 |
| 创 60 日新低家数 | close = min(close, 60d) | 恐慌深度 |
| 市场波动率 (5/10/20 日) | 全市场日收益率标准差 | 波动率飙升=恐慌 |
| 缩放量强度 | 全市场成交额 / 20日均成交额 | >1.3 放量, <0.7 缩量 |
| 板块集中度 | 涨幅 Top3 板块涨停股 / 全市场涨停 | 高=抱团, 低=普涨 |

**数据来源**: `stock_daily` 表 + `sector_stock` 表 (均已存在)

### Layer 2: xtdata 补充 (已有通道)

从 QMT 本地缓存补充 Layer 1 无法覆盖的维度。

| 指标 | xtdata 接口 | 含义 |
|------|------------|------|
| 板块实时热度 | `get_stock_list_in_sector()` + 涨幅排序 | 热门板块识别 |
| 大盘指数位置 | `get_market_data_ex(["000001.SH"])` | 均线多空排列 |
| ETF 溢折价 | `get_market_data_ex()` ETF 净值 vs 市价 | 资金态度 |

### Layer 3a: akshare 直调 (免费, 不需要 API Key)

每日定时采集，总请求量极低 (< 15 次/天)，不触发限流。

| 指标 | akshare 接口 | 频率 | 说明 |
|------|-------------|------|------|

## 情绪特征工程

### 原始指标 → 模型特征

对每个原始情绪指标，生成 5 类衍生特征:

```
原始指标 (如 ad_ratio = 1.3)
    │
    ├── 滚动均值: ad_ratio_ma5, ad_ratio_ma10, ad_ratio_ma20
    ├── 偏离度:   ad_ratio_dev = (当前 - MA20) / MA20
    ├── 差分:     ad_ratio_diff = 今日 - 昨日 (变化速度)
    ├── Z-score:  ad_ratio_zscore = (当前 - 60日均) / 60日标准差
    └── 分位数:   ad_ratio_pctile = 在过去 250 日中的百分位
```

### 情绪分类

| 维度 | 指标组合 | 输出 |
|------|---------|------|
| 赚钱效应 | 涨跌比 + 涨停数 + 创新高数 | earning_effect: -1 ~ +1 |
| 资金情绪 | 北向净流入 + 融资余额变化 | capital_mood: -1 ~ +1 |
| 波动情绪 | 市场波动率 + 跌停数 + 缩放量 | volatility_mood: -1 ~ +1 |
| 板块热度 | 板块集中度 + 热门板块数 | sector_heat: -1 ~ +1 |
| 新闻情绪 | OpenClaw 新闻评分 + 事件影响 | news_mood: -1 ~ +1 |
| 外围情绪 | 汇率变动 + 黄金 + 原油 | global_mood: -1 ~ +1 |

### 合成情绪指数 (Composite Sentiment Index)

将 6 个维度加权合成为单一指数:

```
CSI = w1 × earning_effect     (权重 0.25, 最直接)
    + w2 × capital_mood        (权重 0.25, 最准确)
    + w3 × volatility_mood     (权重 0.15, 前瞻)
    + w4 × sector_heat         (权重 0.10, 结构)
    + w5 × news_mood           (权重 0.15, 事件驱动)
    + w6 × global_mood         (权重 0.10, 外围)
```

输出范围: **-1.0 (极度恐慌) ~ +1.0 (极度贪婪)**

权重可通过 `.env` 配置调整。

**`null`** = 使用 .env 中的默认参数 (不覆盖)。如果某策略在某状态下不出现在 `preferred_strategies` 中，则不会被执行，Profile 中也不需要该状态的配置。

### Profile 加载逻辑

```python
def get_strategy_config(strategy_name: str, macro_state: str) -> dict:
    """获取策略在当前宏观状态下的参数"""
    profiles = load_profiles("strategy_profiles.json")
    
    if strategy_name not in profiles:
        return {}  # 无 Profile, 使用 .env 默认值
    
    state_config = profiles[strategy_name].get(macro_state)
    
    if state_config is None:
        return {}  # null 或缺失, 使用 .env 默认值
    
    return state_config  # 覆盖 .env 默认值
```

Orchestrator 调用时:

```python
# orchestrator.py
profile = get_strategy_config(alloc.strategy_name, current_state)
base_config = _default_config_from_settings()  # .env 默认值
merged_config = {**base_config, **profile}      # Profile 覆盖
strat = registry.create(alloc.strategy_name, config=merged_config)
```


### sentiment_daily — 每日情绪汇总 (主表)

```sql
CREATE TABLE sentiment_daily (
    trade_date     DATE PRIMARY KEY,
    
    -- ═══════════════════════════════════════════════════
    -- 固定结构字段 (高频查询/聚合/LGB 特征直接使用)
    -- ═══════════════════════════════════════════════════
    
    -- Layer 1: 量价情绪 (从 stock_daily 计算, 零网络成本)
    ad_ratio              FLOAT,    -- 涨跌比 (>1.5乐观, <0.5恐慌)
    limit_up_count        INT,      -- 涨停家数
    limit_down_count      INT,      -- 跌停家数
    burst_rate            FLOAT,    -- 炸板率 (0~1, 高=封板质量差)
    new_high_60d          INT,      -- 60日新高家数
    new_low_60d           INT,      -- 60日新低家数
    market_volatility_5d  FLOAT,    -- 5日市场波动率
    market_volatility_20d FLOAT,    -- 20日市场波动率
    volume_ratio          FLOAT,    -- 缩放量强度 (>1.3放量, <0.7缩量)
    sector_concentration  FLOAT,    -- 板块集中度 (高=抱团, 低=普涨)
    
    -- Layer 3a: akshare 资金数据
    north_net_flow        FLOAT,    -- 北向净流入 (亿元)
    north_cumulative      FLOAT,    -- 北向年内累计净流入 (亿元)
    margin_balance        FLOAT,    -- 融资余额 (亿元)
    margin_balance_change FLOAT,    -- 融资余额日变化 (亿元)
    
    -- Layer 3a: akshare 衍生品/宏观
    futures_basis         FLOAT,    -- 股指期货基差 (%, 正=升水乐观)
    fx_usdcny             FLOAT,    -- 美元兑人民币汇率
    
    -- Layer 3b: OpenClaw/LLM 数值结果
    news_sentiment_score  FLOAT,    -- 新闻情绪评分 (-1~+1)
    gold_price_usd        FLOAT,    -- 黄金价格 (USD/oz)
    crude_oil_usd         FLOAT,    -- 原油价格 (USD/bbl)
    xueqiu_sentiment      FLOAT,    -- 雪球社区情绪 (-1~+1)
    
    -- 合成指标 (特征工程计算后回写)
    earning_effect        FLOAT,    -- 赚钱效应指数 (-1~+1)
    capital_mood          FLOAT,    -- 资金情绪指数 (-1~+1)
    volatility_mood       FLOAT,    -- 波动情绪指数 (-1~+1)
    sector_heat           FLOAT,    -- 板块热度指数 (-1~+1)
    news_mood             FLOAT,    -- 新闻情绪指数 (-1~+1)
    global_mood           FLOAT,    -- 外围情绪指数 (-1~+1)
    composite_sentiment   FLOAT,    -- 合成情绪指数 (-1~+1)
    
    -- 宏观状态判定
    suggested_state       VARCHAR(32),  -- 系统建议的宏观状态
    applied_state         VARCHAR(32),  -- 实际生效的宏观状态
    state_confidence      FLOAT,        -- 分类置信度 (0~1)
    
    -- ═══════════════════════════════════════════════════
    -- JSONB 半结构化字段 (格式多变/嵌套/扩展性强)
    -- ═══════════════════════════════════════════════════
    
    hot_sectors           JSONB DEFAULT '[]'::JSONB,
    -- 热门板块列表, 格式: ["半导体", "AI算力", "军工"]
    -- 每次采集可能板块数量不同, 用 JSONB 数组灵活存储
    
    key_events            JSONB DEFAULT '[]'::JSONB,
    -- 关键事件列表, 格式:
    -- [
    --   {"event": "央行降准50bp", "impact": "positive", "magnitude": "high"},
    --   {"event": "中美贸易谈判", "impact": "neutral", "magnitude": "medium"}
    -- ]
    -- 每天事件数量和结构不固定, 适合 JSONB
    
    hot_stocks            JSONB DEFAULT '[]'::JSONB,
    -- 热门个股列表, 格式:
    -- [
    --   {"code": "601398.SH", "reason": "降准利好银行", "sentiment": 0.8},
    --   {"code": "000001.SZ", "reason": "雪球讨论量飙升", "sentiment": 0.4}
    -- ]
    
    extra                 JSONB DEFAULT '{}'::JSONB,
    -- 扩展字段, 存放不固定的附加信息:
    -- {
    --   "market_mood_text": "央行降准超预期, 市场偏乐观",
    --   "etf_premium_300": 0.12,
    --   "pcr_ratio": 0.85,
    --   "ivix": 18.5,
    --   "custom_field": "任意扩展"
    -- }
    -- 新增数据源的数据可以先存这里, 无需改表结构
    
    created_at  TIMESTAMPTZ DEFAULT NOW(),
    updated_at  TIMESTAMPTZ DEFAULT NOW()
);

-- ═══════════════════════════════════════════════════
-- 索引设计
-- ═══════════════════════════════════════════════════

-- 时间范围查询 (主键已覆盖等值查询)
-- PRIMARY KEY (trade_date) 自带 B-tree 索引

-- JSONB GIN 索引: 加速 JSONB 内部字段查询
-- GIN = Generalized Inverted Index, 适合 JSONB 的 @>, ?, ?| 操作符
CREATE INDEX idx_sentiment_hot_sectors ON sentiment_daily USING GIN (hot_sectors);
CREATE INDEX idx_sentiment_key_events ON sentiment_daily USING GIN (key_events);
CREATE INDEX idx_sentiment_hot_stocks ON sentiment_daily USING GIN (hot_stocks);

-- 宏观状态查询 (找某个状态下的所有日期)
CREATE INDEX idx_sentiment_state ON sentiment_daily (applied_state);

-- 合成指数范围查询 (找情绪极端的日期)
CREATE INDEX idx_sentiment_composite ON sentiment_daily (composite_sentiment);
```

**为什么 hot_sectors / key_events / hot_stocks 用 JSONB 而不是关联表?**

| 方案 | 优点 | 缺点 |
|------|------|------|
| 关联表 (event_id → events 表) | 严格范式化 | 3 张额外表, 多次 JOIN, 开发复杂 |
| JSONB 内嵌 | **一条 SQL 返回全部数据, 查询简单** | 不适合频繁更新内部元素 |

对于我们的场景 (每日写入一次, 读取为主), JSONB 内嵌是最优选择。

## JSONB 查询手册 (实用 SQL 速查)

以下查询可以直接在 PostgreSQL 客户端、DBeaver、或 API 端点中使用。

### 基础查询

```sql
-- 1. 查某天的全部情绪数据
SELECT * FROM sentiment_daily WHERE trade_date = '2026-04-03';

-- 2. 查最近 7 天的合成情绪指数趋势
SELECT trade_date, composite_sentiment, applied_state
FROM sentiment_daily
WHERE trade_date >= CURRENT_DATE - INTERVAL '7 days'
ORDER BY trade_date;

-- 3. 查情绪极端的日期 (恐慌或贪婪)
SELECT trade_date, composite_sentiment, applied_state
FROM sentiment_daily
WHERE composite_sentiment > 0.7 OR composite_sentiment < -0.7
ORDER BY composite_sentiment DESC;
```

### JSONB 数组查询

```sql
-- 4. 查某天的热门板块
SELECT trade_date, hot_sectors
FROM sentiment_daily
WHERE trade_date = '2026-04-03';
-- 返回: ["银行", "券商", "地产"]

-- 5. 查哪些天 "半导体" 是热门板块
--    @> 操作符: JSONB 包含判断 (需要 GIN 索引加速)
SELECT trade_date, hot_sectors, composite_sentiment
FROM sentiment_daily
WHERE hot_sectors @> '"半导体"'::JSONB
ORDER BY trade_date DESC;

-- 6. 查哪些天热门板块中包含 "银行" 或 "券商"
--    ?| 操作符: 数组中存在任一元素
SELECT trade_date, hot_sectors
FROM sentiment_daily
WHERE hot_sectors ?| ARRAY['银行', '券商']
ORDER BY trade_date DESC
LIMIT 20;
```

### JSONB 对象数组查询

```sql
-- 7. 查包含 "高影响正面事件" 的日期
SELECT trade_date, key_events
FROM sentiment_daily
WHERE key_events @> '[{"impact": "positive", "magnitude": "high"}]'::JSONB;
-- 这会找到 key_events 数组中任意元素同时满足 impact=positive 且 magnitude=high 的行

-- 8. 查某只股票曾被标记为热门的所有记录
SELECT trade_date, hot_stocks, composite_sentiment
FROM sentiment_daily
WHERE hot_stocks @> '[{"code": "601398.SH"}]'::JSONB
ORDER BY trade_date DESC;

-- 9. 展开 key_events 数组, 逐事件分析
SELECT
    s.trade_date,
    event_item->>'event' AS event_desc,
    event_item->>'impact' AS impact,
    event_item->>'magnitude' AS magnitude
FROM sentiment_daily s,
     JSONB_ARRAY_ELEMENTS(s.key_events) AS event_item
WHERE s.trade_date >= '2026-03-01'
  AND event_item->>'magnitude' = 'high'
ORDER BY s.trade_date DESC;

-- 10. 统计每个板块出现热门的次数 (板块热度排行)
SELECT sector, COUNT(*) AS hot_days
FROM sentiment_daily,
     JSONB_ARRAY_ELEMENTS_TEXT(hot_sectors) AS sector
WHERE trade_date >= CURRENT_DATE - INTERVAL '30 days'
GROUP BY sector
ORDER BY hot_days DESC
LIMIT 10;
```

### JSONB + 结构化字段混合查询

```sql
-- 11. JSONB 与其他表 JOIN (MongoDB 做不到的核心优势)
--     查热门股票的实际涨跌幅
SELECT
    s.trade_date,
    hs->>'code' AS stock_code,
    hs->>'reason' AS hot_reason,
    (hs->>'sentiment')::FLOAT AS hot_sentiment,
    d.close,
    d.pct_chg
FROM sentiment_daily s,
     JSONB_ARRAY_ELEMENTS(s.hot_stocks) AS hs
JOIN stock_daily d
    ON d.code = hs->>'code'
    AND d.trade_date = s.trade_date
WHERE s.trade_date = '2026-04-03';

-- 12. 查北向资金大幅流入 + 新闻情绪乐观的日期 (混合查询)
SELECT trade_date, north_net_flow, news_sentiment_score,
       hot_sectors,
       extra->>'market_mood_text' AS mood
FROM sentiment_daily
WHERE north_net_flow > 50
  AND news_sentiment_score > 0.5
ORDER BY north_net_flow DESC;

-- 13. 查 extra 扩展字段中的 ETF 溢价率
SELECT trade_date, 
       (extra->>'etf_premium_300')::FLOAT AS etf_premium
FROM sentiment_daily
WHERE extra ? 'etf_premium_300'  -- 只查有这个字段的行
  AND (extra->>'etf_premium_300')::FLOAT > 0.5
ORDER BY trade_date DESC;
```

### 采集日志查询

```sql
-- 14. 查某天所有采集记录
SELECT source_name, schedule_slot, status, llm_provider,
       llm_tokens_in, llm_tokens_out, llm_cost_cny
FROM sentiment_ingest_log
WHERE trade_date = '2026-04-03'
ORDER BY collected_at;

-- 15. 查 LLM 清洗的月度成本统计
SELECT
    DATE_TRUNC('month', trade_date) AS month,
    COUNT(*) AS total_calls,
    SUM(llm_tokens_in) AS total_tokens_in,
    SUM(llm_tokens_out) AS total_tokens_out,
    SUM(llm_cost_cny) AS total_cost_cny
FROM sentiment_ingest_log
WHERE llm_provider IS NOT NULL
GROUP BY DATE_TRUNC('month', trade_date)
ORDER BY month DESC;

-- 16. 查失败的采集记录 (排查问题)
SELECT trade_date, source_name, status, error_message,
       raw_data
FROM sentiment_ingest_log
WHERE status != 'success'
ORDER BY created_at DESC
LIMIT 20;

-- 17. 查某次采集的原始数据 vs 清洗结果 (对比审计)
SELECT
    source_name,
    raw_data,      -- 原始: 可能是长文本、HTML 等
    cleaned_data   -- 清洗后: 标准 JSON
FROM sentiment_ingest_log
WHERE trade_date = '2026-04-03'
  AND source_name = 'openclaw_news';
```

### JSONB 操作符速查表

| 操作符 | 含义 | 示例 |
|--------|------|------|
| `->` | 取 JSON 对象的键 (返回 JSONB) | `extra->'market_mood_text'` |
| `->>` | 取 JSON 对象的键 (返回 TEXT) | `extra->>'market_mood_text'` |
| `@>` | 左边包含右边 | `hot_sectors @> '"银行"'::JSONB` |
| `?` | 对象中是否存在某个键 | `extra ? 'etf_premium_300'` |
| `?|` | 数组中存在任一元素 | `hot_sectors ?| ARRAY['银行','券商']` |
| `?&` | 数组中全部存在 | `hot_sectors ?& ARRAY['银行','券商']` |
| `||` | JSONB 合并 | `extra || '{"new_key":"val"}'` |
| `#>` | 按路径取 JSONB | `key_events#>'{0,event}'` |
| `#>>` | 按路径取 TEXT | `key_events#>>'{0,event}'` |

> **提示**: 对 JSONB 列创建 GIN 索引后, `@>`, `?`, `?|`, `?&` 操作符都会走索引, 性能优秀。

```python
# src/sentiment/scheduler.py

from apscheduler.schedulers.asyncio import AsyncIOScheduler
from apscheduler.triggers.cron import CronTrigger

scheduler = AsyncIOScheduler()

def setup_sentiment_scheduler():
    """注册所有情绪采集定时任务"""
    
    # 根据 .env 配置获取采集时间点
    # SENTIMENT_COLLECT_SCHEDULE=09:00,12:00,15:30,20:00
    schedules = settings.sentiment.collect_schedule.split(",")
    slot_names = ["pre_market", "midday", "post_market", "evening"]
    
    for time_str, slot in zip(schedules, slot_names):
        hour, minute = time_str.strip().split(":")
        scheduler.add_job(
            run_collect,
            trigger=CronTrigger(
                hour=int(hour),
                minute=int(minute),
                day_of_week="mon-fri",  # 仅交易日
                timezone="Asia/Shanghai",
            ),
            kwargs={"schedule_slot": slot},
            id=f"sentiment_{slot}",
            name=f"情绪采集: {slot}",
            replace_existing=True,
        )
    
    # 每日 16:00 计算合成指数和宏观分类
    scheduler.add_job(
        run_compute_and_classify,
        trigger=CronTrigger(hour=16, minute=0, day_of_week="mon-fri"),
        id="sentiment_classify",
        name="合成指数计算 + 宏观分类",
        replace_existing=True,
    )
    
    # 每日 02:00 清理 30 天前的日志
    scheduler.add_job(
        cleanup_old_logs,
        trigger=CronTrigger(hour=2, minute=0),
        id="sentiment_cleanup",
        name="清理过期采集日志",
        replace_existing=True,
    )


async def run_collect(schedule_slot: str):
    """执行一次完整的情绪数据采集"""
    logger.info(f"开始情绪采集: {schedule_slot}")
    
    # 1. 查注册表, 获取当前时间点应该采集哪些数据源
    sources = registry.get_sources_for_slot(schedule_slot)
    
    for source in sources:
        try:
            if source.type == "local_db":
                raw = await collect_price_volume(source)
            elif source.type == "akshare":
                raw = await collect_akshare(source)
            elif source.type == "openclaw_push":
                continue  # 被动接收, 不主动采集
            elif source.type == "tavily_search":
                raw = await collect_tavily(source)
            else:
                continue
            
            # 2. 清洗 (结构化直通, 非结构化走 LLM)
            cleaned = await cleaner.process(source.name, raw)
            
            # 3. 入库
            await save_to_db(source, raw, cleaned)
            
        except Exception as e:
            # 记录失败日志, 不中断其他数据源
            await log_failure(source, str(e))
            logger.warning(f"采集失败 [{source.name}]: {e}")
    
    logger.info(f"情绪采集完成: {schedule_slot}")


async def run_compute_and_classify():
    """计算合成指数 + 宏观状态分类"""
    today = date.today()
    
    # 1. 特征工程: 计算滚动均值/Z-score/偏离度
    features = await feature_builder.compute(today)
    
    # 2. 合成情绪指数
    composite = await composite_index.calculate(features)
    
    # 3. 宏观状态分类
    result = await macro_classifier.classify(features, composite)
    
    # 4. 回写数据库
    await update_sentiment_daily(today, features, composite, result)
    
    # 5. 通知 (飞书/日志)
    if result.state_changed:
        await notify_state_change(result)
```

## 热门股 / 一日游识别 (进阶)

利用情绪引擎的数据构建热门股量化指标:

| 指标 | 计算方式 | 说明 |
|------|---------|------|
| RPS 相对强度 | 个股涨幅排名 / 全市场总数 × 100 | ≥85 = 强势股 |
| 涨停频率 | 近 3 日涨停次数 | ≥2 = 连板 |
| 换手率热度 | 换手率 / 20 日均换手率 | ≥2x = 过热 |
| 封板质量 | 封成比 × (1 - 炸板率) | <0.5 = 一日游风险 |
| 板块联动 | 所属板块涨停家数 | <3 = 孤掌难鸣 |

一日游预警信号:
- 量化席位主导 + 无一字板 + 次日回撤 > 1.5% → 一日游概率 > 80%
- 可训练 LGB 二分类模型 (目标: 次日跌幅 ≥ 2% = 一日游)

### schema.py — Pydantic 输出 Schema

```python
from pydantic import BaseModel, Field
from typing import List, Optional

class KeyEvent(BaseModel):
    event: str = Field(description="事件描述")
    impact: str = Field(description="positive/negative/neutral")
    magnitude: str = Field(description="high/medium/low")

class HotStock(BaseModel):
    code: str = Field(description="股票代码, QMT格式如000001.SZ")
    reason: str = Field(description="被关注的原因")
    sentiment: float = Field(ge=-1, le=1, description="情绪评分")

class SentimentExtraction(BaseModel):
    """OpenClaw/LLM 清洗输出的标准格式"""
    news_sentiment_score: float = Field(ge=-1, le=1)
    hot_sectors: List[str] = Field(max_length=5)
    key_events: List[KeyEvent] = Field(max_length=10)
    hot_stocks: List[HotStock] = Field(max_length=10)
    gold_price_usd: Optional[float] = None
    crude_oil_usd: Optional[float] = None
    fx_usdcny: Optional[float] = None
    market_mood_text: str = Field(max_length=100)
```

用 Pydantic 做 **双重校验**: LLM 输出 JSON → Pydantic 解析验证 → 失败则重试一次。

### cleaner.py — 清洗调度逻辑

```python
class SentimentCleaner:
    def process(self, source_name: str, raw_data: Any) -> dict:
        """根据数据源类型选择清洗路径"""
        
        source = registry.get(source_name)
        
        if source.type in ("akshare", "local_db", "xtdata"):
            # 已结构化数据 → 直接映射到 sentiment_daily 字段
            return self._map_structured(source, raw_data)
        
        elif source.type in ("openclaw_push",):
            # OpenClaw 已清洗的 JSON → 校验后入库
            return self._validate_openclaw(raw_data)
        
        else:
            # 非结构化数据 → LLM 清洗
            return self._llm_clean(source, raw_data)
    
    def _llm_clean(self, source, raw_text: str) -> dict:
        """调用 DeepSeek/Qwen 清洗非结构化文本"""
        response = self.llm_client.chat.completions.create(
            model="deepseek-chat",
            response_format={"type": "json_object"},
            messages=[
                {"role": "system", "content": CLEANING_PROMPT},
                {"role": "user", "content": raw_text}
            ],
            temperature=0.1,  # 低温度 = 高确定性
            max_tokens=2000,
        )
        result = json.loads(response.choices[0].message.content)
        # Pydantic 校验
        validated = SentimentExtraction(**result)
        return validated.model_dump()
```

## 常见问题

### 技术选型类

### Q: 为什么不用 Apache Flink?
Flink 是分布式流计算引擎，专为 Level-2 毫秒级数据设计 (日均 30GB+)。qt-quant 是日频散户量化，每天采集 4 次，每次几条数据。用 APScheduler (Python 定时调度器) 内嵌在 FastAPI 中就够了，零额外部署。引入 Flink = 多一套 Kafka + ZooKeeper 集群 + 4-8GB 内存，收益为零。

### Q: 为什么不用 MongoDB?
项目已在用 PostgreSQL。PostgreSQL JSONB 完美平替 MongoDB: 同一张表中结构化字段 (FLOAT) 和半结构化字段 (JSONB) 共存，还能 JOIN `stock_daily` 等关系表 (MongoDB 做不到)。零额外成本，不增加运维负担。

### Q: JSONB 和普通 JSON 有什么区别?
JSONB = JSON Binary，写入时解析为二进制，**查询比 JSON 快很多**。支持 GIN 索引加速 `@>` (包含) 和 `?` (存在) 操作。PostgreSQL 中应始终使用 JSONB 而不是 JSON。唯一缺点: 写入稍慢 (需要解析)，但对日频场景可忽略。

### Q: JSONB 查询性能够吗?
日频场景下完全足够。每天 1 行 `sentiment_daily`，一年 250 行，十年 2500 行。加上 GIN 索引和 B-tree 主键索引，任何查询都在毫秒级。百万行以下 PostgreSQL JSONB 无压力。

### Q: 为什么不用 Redis 做缓存?
日频策略不需要缓存层。每天 4 次采集，每次写入 PostgreSQL 耗时 <1 秒。没有高并发读取场景 (只有你一个人用)。引入 Redis 只增加运维复杂度，不带来性能提升。

### Q: 什么时候需要升级技术栈?
当你的策略升级到 **分钟线实时交易** 时，可能需要:
- Kafka: 行情数据流式传输
- Redis: 分钟级信号缓存
- ClickHouse: 大规模历史数据 OLAP 分析
- Flink: Level-2 实时特征计算
目前日频阶段全部不需要。

### 数据存储类

### Q: 为什么不把原始数据全部存 PostgreSQL 固定字段?
原始数据格式混乱 (HTML/Markdown/自然语言/表格)，强行建固定字段表会导致字段爆炸、NULL 满天飞、查询困难。正确做法:
- 结构化数据 → 固定 FLOAT/INT 字段 (高效查询)
- 半结构化数据 → JSONB 字段 (灵活存储)
- 原始文本 → `ingest_log.raw_data` JSONB (可追溯，30 天轮转)

### Q: JSONB 数据怎么备份?
和普通 PostgreSQL 表一样，`pg_dump` 自动包含 JSONB 数据。无需特殊处理。

```bash
pg_dump -U qt_user -d qt_db -t sentiment_daily -t sentiment_ingest_log > backup.sql
```

### Q: 如何给 JSONB 新增字段 (不改表结构)?
这正是 JSONB 的核心优势。直接在 Python 代码中写入新的 key 即可:

```python
row.extra = {**row.extra, "pcr_ratio": 0.85, "ivix": 18.5}
session.commit()
```

无需 `ALTER TABLE`，无需数据库迁移，无需重启服务。旧数据中没有这个 key 也不影响查询。

### LLM 清洗类

### Q: LLM 清洗会不会太慢?
不会。DeepSeek V3 处理 5000 字约 1-3 秒。每日 4 次采集，总 LLM 调用时间 < 15 秒。对于日频策略完全足够。

### Q: LLM 清洗会不会太贵?
DeepSeek V3 价格: ¥2/百万 input token。每日 4 次采集约 12000 tokens，费用 ¥0.048/天，**每月约 ¥1.5**。基本免费。可以通过 `ingest_log` 表的 `llm_cost_cny` 字段精确追踪成本。

### Q: LLM 返回的 JSON 格式不对怎么办?
三层保障: ① `response_format: json_object` 强制 JSON 输出; ② Pydantic Schema 校验; ③ 校验失败自动重试一次 (切换备用模型)。如果仍然失败，降级到关键词正则提取，标记 `status="partial"`。

### Q: OpenClaw 端清洗 vs qt-quant 端清洗哪个好?
**OpenClaw 端优先 (C1)**: 它本身就有 LLM，收集完直接结构化推送，最简洁，qt-quant 零 LLM 依赖。**qt-quant 端 (C2) 兜底**: 当 OpenClaw 未部署或推送格式不标准时，qt-quant 自己调 DeepSeek/Qwen API 清洗。两条路径可以并存。

### 运维类

### Q: akshare 被限流了怎么办?
系统有 5 层数据源。Layer 1 (量价情绪) 完全不走网络。Layer 3 的 akshare 如果某个接口失败，会自动 fallback 到 Tavily 搜索或 Browser 提取。每日总请求量 < 25 次，正常情况不会触发限流。

### Q: OpenClaw 没有部署怎么办?
情绪引擎不依赖 OpenClaw。Layer 1 (量价) + Layer 3a (akshare) + C2 (qt-quant 自调 LLM) 就能覆盖全部功能。OpenClaw 是增强项不是必需项。

### Q: 定时任务不执行怎么排查?

```python
# 1. 查看调度器状态
curl http://localhost:8012/api/sentiment/scheduler-status

# 2. 查看最近采集日志
SELECT source_name, schedule_slot, status, error_message, collected_at
FROM sentiment_ingest_log
WHERE trade_date = CURRENT_DATE
ORDER BY collected_at DESC;

# 3. 检查 .env 配置
# SENTIMENT_ENABLED=true
# SENTIMENT_COLLECT_SCHEDULE=09:00,12:00,15:30,20:00
```

### Q: 采集日志会不会撑爆数据库?
不会。每日约 15-20 条日志 (4 个时间点 × 4-5 个数据源)，每条约 2-5KB。一年约 5000 条 ≈ 25MB。另外有定时任务每天凌晨自动清理 30 天前的日志。

### 策略配置类

### Q: 策略参数 Profile 和 .env 参数冲突怎么办?
Profile 覆盖 .env。加载顺序: `.env` 默认值 → `strategy_profiles.json` 覆盖。如果 Profile 中某个参数未指定，使用 `.env` 的值。如果 Profile 值为 `null`，表示该策略在该宏观状态下使用 `.env` 全部默认参数。

### Q: 合成情绪指数的权重怎么调?
通过 `.env` 中 `SENTIMENT_W_*` 系列参数调整。也可以用 LGB 模型自动学习最优权重 (P2 阶段)。

### Q: 家用显卡能跑本地 NLP 吗?
FinBERT (110M 参数) < 2GB 显存，GTX 1060 都够。纯 CPU 也能跑，单批几秒。但有了 DeepSeek/Qwen API (¥1.5/月)，本地 NLP 的必要性大幅降低。P3 阶段可选实现。

### Q: 需要单独搞爬虫吗?
不需要。OpenClaw 的 `agent-browser-clawdbot` 就是一个智能爬虫 (真实浏览器 + LLM 理解页面)。比传统 XPath/CSS 选择器爬虫更灵活，网站改版也不怕。对于没有 OpenClaw 的情况，Tavily 搜索 API (免费 1000 次/月) 也够用。

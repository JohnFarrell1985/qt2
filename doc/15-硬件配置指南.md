# 硬件配置指南

> 最后更新: 2026-04-05
>
> 本文档针对 **A 股量化因子迭代平台** 的不同部署阶段，给出三档硬件配置方案。
> 原则: **够用即可，避免浪费**。先在家用机完成开发和回测验证，后期按需升级。

---

## 一、系统资源消耗分析

在选型之前，先明确各模块对硬件的实际需求:

| 模块 | CPU | 内存 | GPU | 磁盘 I/O | 说明 |
|------|-----|------|-----|---------|------|
| **LightGBM 训练** | ★★★★ | ★★★ | ☆ (可选) | ★★ | 主要吃 CPU 多线程, 50 只股票 × 252 天 × 158 因子不到 1GB 内存 |
| **XGBoost / CatBoost** | ★★★★ | ★★★ | ★★ (可选) | ★★ | XGBoost GPU 加速约 5x, CatBoost 原生 GPU 支持 |
| **Alpha158 因子计算** | ★★★ | ★★★ | ☆ | ★★★ | pandas 向量化运算, 全 A 股 5000+ 支的话峰值约 8-12 GB 内存 |
| **回测引擎** | ★★★ | ★★ | ☆ | ★★ | 纯 CPU 计算, 日频回测内存占用很低 |
| **PostgreSQL 16** | ★★★ | ★★★ | ☆ | ★★★★ | 数据量约 5-20 GB (日线+分钟线), SSD 对查询性能影响大 |
| **FastAPI 服务** | ★★ | ★ | ☆ | ★ | 轻量 Web 服务, 资源需求很低 |
| **数据采集 (curl_cffi/Playwright)** | ★★ | ★★ | ☆ | ★ | 网络 I/O 密集, Playwright 浏览器实例约 300-500 MB/个 |
| **LLM API 调用 (DeepSeek/Qwen)** | ★ | ★ | ☆ | ☆ | 仅 HTTP 请求, 推理在云端 |
| **知识蒸馏 LoRA 微调** | ★★★ | ★★★ | ★★★★★ | ★★★ | 核心 GPU 消耗, FinBERT2-Base ~4GB, Qwen3-0.6B ~6GB |
| **蒸馏模型推理 (ONNX)** | ★★ | ★ | ★★ (可选) | ★ | ONNX Runtime CPU 推理约 2-5ms/条, GPU 可降至 <1ms |
| **FinBERT2 本地推理** | ★★ | ★★ | ★★ (可选) | ★ | 125M 参数, ONNX INT8 CPU ~5ms, GPU ~2ms |
| **ETF 轮动 / 组合优化 (skfolio)** | ★★ | ★★ | ☆ | ★ | cvxpy 求解器 CPU 密集但规模小 (20-30 ETF) |
| **MLflow 实验管理** | ★ | ★★ | ☆ | ★★ | 主要是元数据存储和 artifact logging |

> ★ = 低需求, ★★★★★ = 高需求, ☆ = 不需要

## 二、方案一: 家用开发机 (入门/开发/回测)

**适用场景**: 个人量化爱好者, 日常开发调试, 小规模回测 (单策略、少量标的), 不做本地 LLM 微调。

**总预算**: 6,000 - 12,000 元 (已有电脑可跳过, 主要看内存和 SSD)

### 2.1 推荐配置

| 硬件 | 推荐规格 | 说明 |
|------|---------|------|
| **CPU** | AMD Ryzen 7 7800X3D (8C/16T) 或 Intel i7-14700K (20C/28T) | LightGBM 训练和因子计算主要吃多线程, 8 核以上即可 |
| **内存** | **32 GB** DDR5 (2×16GB) | 全 A 股因子计算峰值约 8-12GB, pandas + PostgreSQL + IDE 日常约 12-16GB |
| **GPU** | **集成显卡即可** (不做 LLM 微调) 或 RTX 4060 Ti 16GB (选配) | LightGBM 主要用 CPU; RTX 4060 Ti 可应对未来 FinBERT2/Qwen3-0.6B LoRA 微调 |
| **系统盘** | NVMe SSD 512 GB (PCIe 4.0) | 装系统 + PostgreSQL + 代码, 随机读写对 DB 影响大 |
| **数据盘** | SATA SSD 1 TB 或 NVMe 1 TB | 历史行情数据 (日线约 2GB, 分钟线约 15-30GB, 预留扩展) |
| **网络** | 100 Mbps 以上家庭宽带 | 数据下载和 LLM API 调用需稳定网络 |
| **电源** | 550W - 650W 80 Plus 铜牌 | 无独显或轻量独显足够 |
| **OS** | Windows 10/11 Pro 或 Ubuntu 22.04+ | QMT 需要 Windows; 纯开发可用 Linux |

### 2.2 PostgreSQL 调优参数 (32 GB 内存)

```ini
shared_buffers = 8GB
effective_cache_size = 24GB
work_mem = 64MB
maintenance_work_mem = 1GB
max_connections = 50
wal_buffers = 64MB
```

### 2.3 能跑什么

| 功能 | 是否可行 | 备注 |
|------|---------|------|
| 全 A 股日频因子计算 (Alpha158) | 可以 | 约 3-5 分钟完成 5000 只股票 |
| LightGBM 训练 (50-500 只股票池) | 可以 | 单次训练 10-60 秒 |
| XGBoost / CatBoost 训练 | 可以 (CPU) | GPU 加速需独显 |
| 日频回测 (5 年) | 可以 | 单策略 < 30 秒 |
| 分钟频回测 | 勉强 | 大标的池内存可能紧张 |
| LLM 数据清洗 (API) | 可以 | 通过 DeepSeek/Qwen API, 不需本地 GPU |
| LoRA 微调 FinBERT2-Base | 需 RTX 4060 Ti | 125M 参数, ~4GB 显存 |
| LoRA 微调 Qwen3-0.6B | 需 RTX 4060 Ti | 600M 参数, ~6GB 显存 (QLoRA ~4GB) |
| 数据采集 (Playwright) | 可以 | 建议限制并发浏览器实例 ≤ 3 |
| QMT 实盘交易 | 可以 | QMT 本身资源占用很低 |

### 2.4 不能/不建议做的

- 本地运行 7B+ 大模型推理 (显存不够)
- 大规模分钟线回测 (内存瓶颈, 建议上 64GB)
- 多模型并行训练 (CPU 和内存瓶颈)

## 三、方案二: 云服务器 (生产/多策略/7×24 运行)

**适用场景**: 策略验证完毕准备上线, 需要 7×24 稳定运行, 多策略并行回测, 定时数据采集和模型重训练, 小规模 LoRA 微调。

**月费预估**: 1,500 - 4,000 元/月 (国内云厂商)

### 3.1 推荐配置

| 硬件 | 推荐规格 | 云厂商参考 |
|------|---------|-----------|
| **CPU** | 16 核 vCPU (AMD EPYC / Intel Xeon) | 阿里云 ecs.g7.4xlarge / 腾讯云 SA5.4XLARGE |
| **内存** | **64 GB** | 多策略并行 + PostgreSQL + 数据采集同时运行 |
| **GPU** | **RTX 4090 单卡** (24 GB) 或 **A10 单卡** (24 GB) | 阿里云 gn7i (A10) / AutoDL RTX 4090 |
| **系统盘** | NVMe SSD 100 GB | 系统 + 应用 |
| **数据盘** | 高效云盘 500 GB - 1 TB | PostgreSQL 数据 + 模型文件 + 日志 |
| **网络** | 5 Mbps+ 带宽 (按流量另选更经济) | 内网访问 PostgreSQL 无带宽限制 |
| **OS** | Ubuntu 22.04 LTS / Debian 12 | 生产推荐 Linux |

### 3.2 GPU 选型建议

| GPU | 显存 | 月租参考 | 适用范围 |
|-----|------|---------|---------|
| **无 GPU** | — | 800-1,200 元 | 仅 LightGBM + 回测 + API 服务 (不做微调) |
| **RTX 4090** | 24 GB | 1,500-2,500 元 | LoRA 微调 (FinBERT2-Base/Qwen3-0.6B) + ONNX/GGUF 推理 |
| **A10** | 24 GB | 1,200-2,000 元 | 与 RTX 4090 类似, 更适合 7×24 长期运行 (ECC 显存) |
| **A800 80GB** | 80 GB | 4,000-6,000 元 | 多模型并行推理, 大 batch, 可微调 7B+ 模型 |
| **A100 40GB** | 40 GB | 3,000-4,500 元 | 可微调 3B-7B 模型, 但对本项目偏奢侈 |
| **A100 80GB** | 80 GB | 5,000-7,000 元 | 多模型并行, 大规模蒸馏标注, 企业级 |

> **性价比推荐**: 如果不做 LLM 微调, 选无 GPU 方案即可, LightGBM 不需要 GPU。
> 如果需要 LoRA 微调, RTX 4090 是最佳性价比选择。
> 如果团队有 A800/A100 资源, 代码无需修改, 仅需调整 `.env` 配置即可自动适配 (详见第九章)。

### 3.3 PostgreSQL 调优参数 (64 GB 内存)

```ini

work_mem = 128MB
maintenance_work_mem = 2GB
max_connections = 100
wal_buffers = 128MB

```

### 3.4 能跑什么

| 功能 | 是否可行 | 备注 |
|------|---------|------|
| 全部 方案一 功能 | 可以 | 速度更快, 可长期稳定运行 |
| 多策略并行回测 (3-5 策略) | 可以 | 16 核 CPU 支撑多线程 |
| 大规模分钟线回测 | 可以 | 64 GB 内存足够 |
| 定时数据采集 + LLM 清洗 | 可以 | APScheduler 7×24 运行 |
| LoRA 微调 FinBERT2-Base / Qwen3-0.6B | 可以 (需 GPU) | RTX 4090 24GB 绰绰有余 |
| LoRA 微调 Qwen2.5-0.5B | 可以 (需 GPU) | ~6GB 显存, QLoRA 可降至 ~4GB |
| ONNX Runtime 推理 (蒸馏模型) | 可以 | CPU 推理 2-5ms, GPU <1ms |
| ETF 轮动 + skfolio 组合优化 | 可以 | 计算量小 |
| MLflow 实验管理 | 可以 | 推荐用同机 PostgreSQL 存元数据 |
| QMT 实盘交易 | 需 Windows | 建议另起一台 Windows 云主机或本地机运行 QMT |

### 3.6 省钱技巧

- **竞价实例 / Spot**: 阿里云竞价实例可节省 50-80%, 适合离线回测和训练任务
- **GPU 按需开关**: 训练时开 GPU 实例, 平时用 CPU 实例, 可节省 60%+ GPU 费用
- **AutoDL / 矩池云**: 国内 GPU 云平台, RTX 4090 时租约 2-4 元/小时, 适合间歇性训练
- **数据盘快照**: 定期创建快照而非全量备份, 存储费用更低
- **分离部署**: PostgreSQL 用低配长期实例, GPU 训练用按需短期实例

## 四、方案三: 裸金属服务器 (全功能/大规模/团队)

**适用场景**: 多人团队协作, 大规模因子挖掘 (FactorEngine), 7B-13B 模型 LoRA 微调, 多策略多品种全 A 覆盖, 分钟级高频数据处理。

**一次性投入**: 50,000 - 200,000 元 (不含 GPU); GPU 另算

### 4.1 推荐配置

| 硬件 | 推荐规格 | 说明 |
|------|---------|------|
| **CPU** | AMD EPYC 9454 (48C/96T) 或 Intel Xeon w9-3475X (36C/72T) | 因子挖掘和多策略并行需要大量核心 |
| **内存** | **128 GB** DDR5 ECC (可扩展至 256 GB) | 全 A 股分钟线加载 + 多任务并行 |
| **GPU** | **NVIDIA RTX 4090 × 2** (48 GB 总显存) | 性价比最优: 单卡微调, 双卡并行训练+推理 |
| **系统盘** | NVMe SSD 1 TB (PCIe 5.0) | 系统 + 应用 + 模型文件 |
| **数据盘** | NVMe SSD 2 TB × 2 (RAID 1) | PostgreSQL 数据, RAID 1 保证数据安全 |
| **冷存储** | HDD 4 TB × 2 (RAID 1) | 历史行情归档 + 备份 |
| **网络** | 万兆以太网 (10 GbE) | 内网数据传输, 对接券商 |
| **电源** | 1200W 80 Plus 金牌 | 双 RTX 4090 满载约 900W |
| **散热** | 塔式风冷 (Noctua NH-D15) 或 240mm 水冷 | 48 核 CPU 建议水冷 |
| **UPS** | 1500VA 在线式 UPS | 保护交易连续性, 约 15-20 分钟续航 |
| **OS** | Ubuntu 22.04 LTS Server | 稳定 + 5 年长期支持 |

### 4.2 GPU 选型对比

本项目的 GPU 需求主要集中在 **知识蒸馏 LoRA 微调** 和 **蒸馏模型推理**, 而非训练 70B 大模型。
因此 **不推荐 H100**, 性价比过低。

| GPU | 显存 | 单卡价格 (参考) | 微调能力 | 推理吞吐 | 推荐度 |
|-----|------|----------------|---------|---------|--------|
| **RTX 4090** | 24 GB GDDR6X | ~12,000 元 | FinBERT2-Base / Qwen3-0.6B | ~95 tok/s (7B) | ★★★★★ |
| **RTX 5090** | 32 GB GDDR7 | ~16,000 元 | 同上 + 更大模型余量 | ~130 tok/s (7B) | ★★★★ |
| **A6000** | 48 GB GDDR6 | ~30,000 元 | 可微调 7B-13B LoRA | ~60 tok/s (7B) | ★★★ |
| **A800 80GB** | 80 GB HBM2e | ~50,000 元 | 可微调 13B-30B LoRA, 多模型并行 | ~80 tok/s (7B) | ★★★ |
| **A100 80GB** | 80 GB HBM2e | ~70,000 元 | 可微调 13B-30B LoRA | ~85 tok/s (7B) | ★★ |
| **H100 80GB** | 80 GB HBM3 | ~200,000+ 元 | 可微调 70B LoRA | ~150 tok/s (7B) | ★ |

**结论**: 本项目蒸馏目标是 125M ~ 600M 参数的小模型 (FinBERT2-Base 125M / Qwen3-0.6B 600M), RTX 4090 的 24GB 显存完全够用, 两张 RTX 4090 总价不到一张 A6000 的价格。生产推理阶段 ONNX/GGUF 量化后, 纯 CPU 即可运行 (无需 GPU)。

> **何时考虑 H100**: 仅当团队需要微调 **70B+ 大模型** 作为教师模型, 或做大规模多教师并行标注时才值得。
> 对于本项目, 教师模型用 DeepSeek/Qwen **API 远程调用** 是更经济的方案。

## 五、分阶段演进路线

```
阶段一 (0-3 月)                阶段二 (3-12 月)              阶段三 (12 月+)
家用开发机                     云服务器                     裸金属 (按需)
─────────────                ──────────────              ─────────────
- 代码开发                    - 7×24 稳定运行              - 全 A 覆盖
- 策略原型验证                 - 多策略并行                 - 团队协作
- 小规模回测                   - 定时数据采集               - 因子挖掘
- LLM API 调用                - LoRA 微调                  - 大规模蒸馏
- QMT 模拟盘                  - 实盘交易                   - 高频数据处理
                              - 飞书告警

预算: 0-12K (一次性)           预算: 1.5-4K/月              预算: 50-200K (一次性)
```

### 推荐策略

1. **先用家用机把策略跑通**: 不要一开始就上云或买服务器。开发阶段的瓶颈是算法和策略逻辑, 不是硬件。
2. **策略验证通过后上云**: 需要 7×24 运行和定时任务时再上云, 先从无 GPU 方案开始。
3. **LoRA 微调按需租 GPU**: 不需要 7×24 占用 GPU, 训练完就释放。AutoDL 时租比月租灵活。
4. **裸金属是锦上添花**: 只有确认策略有效、管理资金规模增长后, 才考虑裸金属。

## 八、常见问题

### Q1: 我是否需要 GPU?

**不一定**。本项目的核心 ML 模型是 LightGBM, 它主要依赖 CPU。GPU 仅在以下场景需要:
- 知识蒸馏 LoRA 微调 (P2 阶段任务)
- 本地运行 FinBERT2 推理
- XGBoost/CatBoost GPU 加速训练

如果使用 DeepSeek/Qwen API 做数据清洗, 并且暂不做知识蒸馏, 完全可以不用 GPU。

### Q2: 32GB 内存够用吗?

**日常开发和小规模回测够用**。但以下场景建议 64GB+:
- 全 A 股 5000+ 支分钟线数据加载
- 多策略并行回测
- Playwright 多实例并发采集

### Q3: H100 值得买吗?

**对本项目不值得**。H100 单卡 20 万+ 元, 主要优势是 80GB HBM3 显存和 NVLink 多卡互联, 适合训练 70B+ 大模型。本项目蒸馏的学生模型最大不超过 500M 参数, RTX 4090 (24GB, 1.2 万元) 完全满足, 性价比高 16 倍。

### Q4: 云服务器还是自建?

| 维度 | 云服务器 | 自建裸金属 |
|------|---------|-----------|
| 初始成本 | 低 (月付) | 高 (一次性) |
| 长期成本 (3 年) | 高 (5-15 万) | 低 (5-20 万, 含电费) |
| 运维 | 云厂商负责 | 自己负责 |
| 弹性 | 随时升降配 | 固定 |
| 适合 | 策略验证期 / 小规模 | 确认盈利后 / 多策略 |

**建议**: 年支出超过 5 万元云费用, 且策略已验证有效, 就值得考虑自建。

### Q5: 分钟线数据量有多大?

| 数据类型 | 标的数 | 时间跨度 | 估算大小 |
|---------|--------|---------|---------|
| 日线 OHLCV | 5000 | 10 年 | ~2 GB |
| 1 分钟线 | 5000 | 1 年 | ~15 GB |
| 1 分钟线 | 5000 | 5 年 | ~75 GB |
| 因子值 (158 因子) | 5000 | 5 年 | ~30 GB |
| 总计 (典型场景) | — | 5 年 | ~120 GB |

磁盘建议按 **3 倍数据量** 预留空间 (索引 + WAL + 临时文件 + 增长)。

## 九、显卡灵活部署方案 (统一代码, 多硬件适配)

> **核心原则**: 同一份推理代码, 通过 `.env` 配置 + 运行时自动探测, 无缝适配从家用 CPU 到 A800 服务器的全场景。

### 9.1 问题: 不同显卡部署有什么区别?

| 维度 | 家用机 (CPU / RTX 4060 Ti) | 云服务器 (RTX 4090 / A10) | 高性能服务器 (A800 / A100) |
|------|---------------------------|--------------------------|---------------------------|
| 显存 | 0 / 16 GB | 24 GB | 80 GB |
| 精度支持 | FP32 / FP16 | FP16 / BF16 (部分) | **BF16 / TF32 原生** |
| 多模型并行 | 不支持 (显存不够) | 勉强 (单模型占满) | **可同时加载多个模型** |
| batch size | 1-4 | 8-32 | **64-256** |
| CUDA 计算能力 | sm_89 (Ada) | sm_89 (Ada) / sm_80 (Ampere) | **sm_80 (Ampere)** |
| NVLink | 无 | 无 | **A800: 400GB/s, A100: 600GB/s** |
| ECC 显存 | 无 | A10 有 | **有 (生产级稳定性)** |

**结论**: 差异在于 batch size、精度、并行数, **不在代码逻辑**。通过配置驱动, 代码完全相同。

### 9.3 运行时自动探测

```python
# src/common/device_config.py (续)

import os, logging

logger = logging.getLogger(__name__)

def detect_gpu() -> str:
    """自动探测 GPU 型号, 返回 profile key"""
    override = os.getenv("DEVICE_PROFILE")
    if override and override in PROFILES:
        logger.info(f"使用手动指定 profile: {override}")
        return override

    try:
        import torch
        if not torch.cuda.is_available():
            return "cpu"
        gpu_name = torch.cuda.get_device_name(0).lower()
        vram_mb = torch.cuda.get_device_properties(0).total_mem // (1024 * 1024)
    except ImportError:
        return "cpu"

    # 按关键字匹配 GPU 型号
    _MAP = [
        ("a100", "a100"), ("a800", "a800"), ("a10", "a10"),
        ("4090", "rtx4090"), ("4060", "rtx4060ti"), ("3090", "rtx4090"),
    ]
    for keyword, profile in _MAP:
        if keyword in gpu_name:
            logger.info(f"探测到 GPU: {gpu_name} ({vram_mb}MB) → profile={profile}")
            return profile

    # 未知 GPU — 按显存猜测
    if vram_mb >= 70000:
        return "a100"
    elif vram_mb >= 20000:
        return "rtx4090"
    elif vram_mb >= 14000:
        return "rtx4060ti"
    return "cpu"


def get_device_profile() -> DeviceProfile:
    """获取当前硬件的推理配置 (单例缓存)"""
    key = detect_gpu()
    profile = PROFILES[key]
    logger.info(f"Device profile: {profile.name} | VRAM={profile.vram_gb}GB | "
                f"precision={profile.precision} | parallel={profile.parallel_models}")
    return profile
```

### 9.4 推理服务适配 — 同一代码, 不同硬件

```python
# src/dataclean/cleaners/distilled_cleaner.py (Phase 2-3)

from src.common.device_config import get_device_profile

class DistilledSentimentCleaner:
    def __init__(self):
        self.profile = get_device_profile()
        self._load_model()

    def _load_model(self):
        if "gpu" in self.profile.backend_finbert2:
            import onnxruntime as ort
            providers = ["CUDAExecutionProvider", "CPUExecutionProvider"]
            sess_opts = ort.SessionOptions()
            self.session = ort.InferenceSession(
                settings.finbert2_model_path + "/model.onnx",
                sess_options=sess_opts, providers=providers,
            )
        else:
            import onnxruntime as ort
            self.session = ort.InferenceSession(
                settings.finbert2_model_path + "/model.onnx",
                providers=["CPUExecutionProvider"],
            )
        from transformers import AutoTokenizer
        self.tokenizer = AutoTokenizer.from_pretrained(settings.finbert2_model_path)

    def predict_batch(self, texts: list[str]) -> list[dict]:
        """批量情绪分类 — batch size 自动适配硬件"""
        batch_size = self.profile.max_batch_finbert2
        results = []
        for i in range(0, len(texts), batch_size):
            batch = texts[i : i + batch_size]
            inputs = self.tokenizer(batch, return_tensors="np",
                                    truncation=True, padding=True, max_length=512)
            logits = self.session.run(None, dict(inputs))[0]
            # ... softmax → label → results.append(...)
            results.extend(self._logits_to_labels(logits))
        return results
```

### 9.5 Qwen3-0.6B 推理后端适配

| 硬件 | 推理后端 | 说明 |
|------|---------|------|
| CPU / RTX 4060 Ti | **llama-cpp-python** | GGUF Q4 量化, ~400MB 内存, 最轻量 |
| RTX 4090 / A10 | **llama-cpp-python** (GPU offload) | GPU 加速, 约 15ms/请求 |
| A800 / A100 | **vLLM** | 连续批处理, 高吞吐, 适合多并发 |

```python
# src/dataclean/cleaners/local_extractor.py

class LocalExtractorCleaner:
    """Qwen3-0.6B 本地结构化抽取 — 自动选择推理后端"""

    def __init__(self):
        self.profile = get_device_profile()

        if self.profile.backend_qwen3 == "vllm":
            from vllm import LLM
            self.engine = LLM(
                model=settings.qwen3_local_model_path,
                dtype=self.profile.precision,
                gpu_memory_utilization=0.3,  # A800 上只用 30% 显存, 留空间给其他模型
                max_model_len=int(settings.qwen3_local_context_length),
            )
        else:
            from llama_cpp import Llama
            n_gpu = -1 if self.profile.vram_gb >= 16 else 0  # 有 GPU 就全部 offload
            self.engine = Llama(
                model_path=settings.qwen3_local_model_path + "/qwen3-0.6b-q4_k_m.gguf",
                n_ctx=int(settings.qwen3_local_context_length),
                n_gpu_layers=n_gpu,
                n_threads=self.profile.qwen3_threads,
                verbose=False,
            )
```

### 9.7 各硬件部署对比速查

| 场景 | 家用机 CPU | 家用机 RTX 4060 Ti | 云 RTX 4090 | 服务器 A800 80GB |
|------|-----------|-------------------|-------------|------------------|
| **DEVICE_PROFILE** | `cpu` (自动) | `rtx4060ti` (自动) | `rtx4090` (自动) | `a800` (自动) |
| FinBERT2 延迟 | ~5ms | ~2ms | ~1ms | **<1ms** |
| FinBERT2 batch | 4 | 32 | 64 | **256** |
| Qwen3 推理后端 | llama-cpp (CPU) | llama-cpp (GPU) | llama-cpp (GPU) | **vLLM** |
| Qwen3 延迟 | ~50ms | ~15ms | ~10ms | **~5ms** |
| Qwen3 batch | 1 | 4 | 8 | **32** |
| 并行模型数 | 1 | 1 | 2 | **4** |
| 精度 | FP32 | FP16 | FP16 | **BF16** |
| 改代码? | **不需要** | **不需要** | **不需要** | **不需要** |
| 改配置? | 默认即可 | 默认即可 | 默认即可 | 默认即可 |

### 9.8 A800 vs A100 差异

| 维度 | A100 80GB | A800 80GB |
|------|-----------|-----------|
| FP64 TFLOPS | 19.5 | 19.5 |
| FP32 TFLOPS | 19.5 | 19.5 |
| TF32 TFLOPS | 156 | 156 |
| FP16 / BF16 TFLOPS | 312 | 312 |
| NVLink 带宽 | 600 GB/s | **400 GB/s** (受限) |
| 单卡推理性能 | 相同 | **相同** |
| 多卡训练 | 更快 (NVLink) | 稍慢 (NVLink 带宽低 33%) |
| 代码差异 | **无** | **无** |
| 本项目影响 | 无 (单卡足够) | 无 (单卡足够) |

> **结论**: A800 是 A100 的国内合规版本, NVLink 带宽受限 (400 vs 600 GB/s), 但 **单卡计算性能完全相同**。
> 本项目的小模型 (125M + 600M) 永远不需要多卡并行训练, 因此 **A800 = A100, 代码零差异**。
